# v0.2 产品规则清单

## 这份文档是干什么的

这份文档不讨论：

- 底层 runtime 怎么实现
- LLM 具体接在哪一层
- 网页、CLI、skill 分别怎么接

这份文档只回答一件事：

`当前这版产品逻辑，哪些规则已经定下来了，哪些只是暂定，哪些边界还没收完。`

它的用途是：

- 帮后续实现不再反复改主线
- 帮网页、CLI、agent 外壳对齐同一套行为
- 帮我们判断现在到底还差什么，才算从 demo 走到可持续使用

## 当前结论

先给一句总判断：

`v0.2 的主线产品逻辑已经基本成型，但还没有完全封板。`

当前更像：

- 一套已经能连续推进的主代理产品规则

还不完全像：

- 一套已经把所有长尾边界都定死的正式产品规范

所以这份文档按三层来写：

1. 已定规则
2. 暂定规则
3. 未定边界

## 一. 已定规则

这些规则已经足够稳定，后续实现默认应直接遵守。

### 1. 产品主目标已经定下

`v0.2` 的目标不是扩更多分析模块，而是把一句模糊草稿连续推进到阶段结论。

这意味着后续所有实现都应该优先服务下面这条主线：

- 接住模糊输入
- 给出当前判断
- 追问最小必要信息
- 根据补充继续推进
- 在合适节点停下或给出阶段结论

### 2. 主线阶段已经定下

当前主线阶段按下面这条顺序走：

- 场景对齐
- 问题定义
- 决策挑战
- 验证设计

补充说明：

- `pre_framing` 是前置收敛动作，不是长期独立主线
- `blocked`、`done`、`deferred` 是运行结果，不是方法阶段

### 3. 追问服务于推进，不服务于完整

不是信息缺了就一定问。

只问：

- 会影响当前阶段判断的
- 会决定能不能进入下一阶段的
- 会改变当前结论方向的

不问：

- 对当前阶段没有影响的细枝末节
- 只是为了“看起来更完整”的补充信息

### 4. 单轮追问数量已经定下

单轮追问上限按下面这套规则执行：

- 默认 1 个
- 只有强相关时到 2 个
- 只有成组前提时到 3 个

超过 3 个，就会明显退回问卷体验。

当前已经明确的典型例子：

- `为什么现在做 + 机会成本`：可以一起问 2 个
- `成功指标 + 基线指标`：可以一起问 2 个
- `产品类型 + 主要平台 + 角色三分法`：可以成组保留 3 个

当前不建议的做法：

- 把互不相关的缺口拼成 2 到 3 个问题一起抛给用户
- 因为“都值得补”就顺手多问

### 5. 回答后只从卡点继续

用户补一句，不代表整轮重新分析一遍。

当前已定规则是：

- 先判断这轮补充命中了哪个卡点
- 更新对应状态
- 只从当前阻塞点或当前阶段继续

这条规则是当前产品逻辑最核心的一条，不能轻易改。

### 6. 用户可以不按系统顺序回答

系统不能要求用户像填表一样回答。

当前默认规则：

- 用户可以先补背景
- 可以先补角色
- 可以直接表达决策倾向
- 可以一句里同时补多个点

系统要负责吸收，不应机械重问原题。

### 7. 决策关口优先级高于追问体验

如果当前已经到了人工决策点：

- 不能为了“继续问更自然”绕开关口
- 不能把阻塞态偷偷放行
- 不能把“该你拍板”改成“系统继续替你分析”

这是最近几轮收口里已经明确保住的规则。

### 8. 半步回答已经是正式行为

当前已经明确：

- 用户可能答到一半
- 系统应优先顺着那半步继续问
- 不应装作没看到，又把原题问一遍

同时也已经定下一个限制：

- 半步追问不能破坏阶段推进和关口控制

### 9. 阶段推进后，上一层半步线索可以被轻承接

当前已经明确的规则不是“永远追着旧问题不放”，而是：

- 如果主线已经推进，系统仍可保留一条轻量提示
- 这条提示只用于说明“刚才那句没有丢”
- 它不能反客为主，压过当前阶段焦点

### 10. 追问策略需要作为正式状态保存

当前已经明确，追问不只是文案问题，还需要有一层正式状态，供不同入口共用。

至少应保留：

- 这轮追问状态是什么
- 这轮最多该问几项
- 这轮采用的是哪类追问策略
- 如果这轮不再追问，停下来的原因是什么

这样后续网页 demo、IDE 壳或其他入口，才能稳定复用同一套追问行为，而不是各自猜。

### 11. 多入口产品行为必须一致

`CLI / agent / 网页 demo` 当前可以长得不一样，但核心行为不能不一样。

至少下面几类行为必须一致：

- 阶段推进
- 追问选择
- 阻塞提示
- 暂缓语义
- 半步承接

### 12. 三类显式选择已经定下

当前已经明确，用户一旦给出清晰选择，系统不能再假装没听见。

这三类显式选择现在都属于正式产品规则：

- 明确说继续产品化
- 明确说先走非产品路径
- 明确说先暂缓

对应动作也已经定下：

- 明确继续产品化：
  如果场景前提已经够，就直接从合适阶段继续；
  如果场景前提还不够，就继续补场景，不允许硬跳过
- 明确先走非产品路径：
  直接转 `stage-block-card`
- 明确暂缓：
  直接转 `stage-block-card`，并落 `deferred`

这里有一条边界也已经定下：

- “我还没想好”“先看看能不能”“再决定要不要做产品”这类犹豫表达，不算显式拍板
- 这类输入仍应保留在 `decision-gate-card`

## 二. 暂定规则

这些规则当前已经有方向，也有实现，但还不适合当成长期冻结规范。

### 1. 追问优先级顺序是暂定的

当前倾向顺序是：

1. 能决定阶段推进的
2. 能缩小问题定义空间的
3. 能改变价值判断的
4. 能决定验证动作是否成立的
5. 其他补充信息

这套顺序现在基本合理，但还需要更多真实案例压一轮。

### 2. 前置收敛触发条件还是暂定

`pre_framing` 当前适合在这些场景触发：

- 输入非常模糊
- 同一句里混着问题、解释和方案
- 存在多条都说得通的理解路径

但这里还没有完全定死：

- 多模糊才算该进
- 什么时候该直接旁路
- 不同行业场景下是否要有不同阈值

### 3. 半步线索保留多久还是暂定

当前已经支持：

- 当前轮次优先追
- 阶段推进后轻提示承接

但还没完全定下：

- 要保留几轮
- 什么时候自动失效
- 什么时候应升级成正式待补项

### 4. 项目背景、工作区记忆、长期偏好的产品心智还是暂定

底层能力已经有了，但用户层规则还没最终定清：

- 哪些信息默认属于项目背景
- 哪些只属于当前案例
- 哪些才算长期偏好
- 用户是否要显式编辑这些层

现在是“能用”，但还不算“心智完全稳定”。

### 5. 网页 demo 的展示权重还是暂定

当前网页已经能看：

- 主卡
- 历史
- 运行态
- carryover 提示

但还没有最后定版：

- 哪块永远放前
- 哪块只在有值时出现
- 哪类信息该弱提示，哪类信息该强提示

## 三. 未定边界

这些问题还没有真正定稿，后续如果继续做产品层收口，优先应该补这里。

### 1. 什么叫“当前已经足够形成阶段结论”

这是现在最值得继续收口的一条。

虽然我们已经有阶段推进规则，但还没把下面这些场景完全定死：

- 什么时候只给 review-card 就够了
- 什么时候还必须继续问
- 什么时候虽然还能问，但不值得再问

### 2. 长会话里，旧线索的产品规则还没定稿

现在已经有：

- case 历史
- 工作区记忆
- runtime 压缩
- carryover 提示

但产品规则还没完全明确：

- 哪些旧线索应该继续显式展示
- 哪些应该只留在历史里
- 哪些应该自动折叠成背景

### 3. “下一步建议”到底是提示还是动作，还没完全定下

当前系统会给：

- 先补什么
- 更建议先补
- 如果继续往下聊，优先补哪几项

但还没完全定稿：

- 这些是不是都算同一级信息
- 是否要收成更少的层级
- 是否要区分“建议动作”和“建议回答”

### 4. 用户主动问“我现在下一步该做什么”时，要不要切特殊模式

当前能接这类输入，但产品规则还没完全定死：

- 是按普通回复处理
- 还是进入一个更明确的“导航回答”
- 这类回答是否应该比普通审查卡更短、更直接

### 5. 真实 agent 感还缺一个更清晰的定义

当前产品已经不只是多轮问答，但离“明显像 agent”还差最后一段定义：

- 用户什么时候会觉得它真的在持续推进
- 哪些行为只是“更自然的问答”
- 哪些行为才算“代理式协作”

这个边界如果不先写清，后面网页、MCP、skill、agent 外壳都会各自理解。

## 四. 当前实施建议

如果按产品收口顺序继续做，建议下一步按下面三件事推进：

### 1. 先冻结这份规则清单

后续新增能力前，先默认以这份文档为准。

如果新实现和规则冲突：

- 先改规则文档
- 再改实现

不要反过来。

### 2. 再补一份“阶段结论判定清单”

最值得继续补的是：

- 什么情况下可以不再追问，直接形成阶段结论
- 什么情况下必须再追一轮
- 什么情况下应明确转人工选择

配套文档可直接看：

- [stage-conclusion-checklist.md](/Users/wannz/Documents/sourcetree/pm-method-agent/docs/stage-conclusion-checklist.md)
- [stage-conclusion-samples.md](/Users/wannz/Documents/sourcetree/pm-method-agent/docs/stage-conclusion-samples.md)

### 3. 最后再回到网页和外壳

网页、skill、agent、MCP 这些外壳后面都要做。

但在这之前，更重要的是：

- 主线规则先稳定
- 用户心智先稳定
- 长尾边界先收住

## 五. 一句话收口

当前 `PM Method Agent` 的产品逻辑已经不再是“想到哪补到哪”。

它已经有了一套可运行、可回归、可继续收窄的主线规则。

接下来最重要的，不是继续铺更多入口，而是把最后那几条关键边界定清，让这套逻辑真正进入“可持续使用”的状态。
