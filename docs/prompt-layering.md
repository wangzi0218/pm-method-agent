# Prompt 分层设计

## 目标

这一层的目标，不是把 prompt 写得更长，而是把 prompt 从“散落在代码里的说明字符串”收敛成可维护的分层结构。

如果不做这一步，后续最容易出现的问题就是：

- 每修一个边界问题，就往 prompt 里再加一句
- 不同入口各写一套 prompt，慢慢漂移
- 团队不知道某条规则到底属于身份、行为、工具约束，还是只是一次性的追加要求

## 当前实现

当前仓库已经补上：

- `src/pm_method_agent/prompting.py`

并已经接入这些 LLM 入口：

- `reply_interpreter`
- `pre_framing`
- `case_copywriter`

这意味着：

- 三个入口不再各自维护一段散乱的 system prompt
- prompt 现在至少有统一结构和最小优先级模型

## 当前分层

第一版先收敛为这几层：

### `身份描述`

说明当前模型是谁、要解决什么问题。

### `角色职责`

说明当前能力边界，例如：

- 只负责解释回复
- 只负责增强候选方向
- 只负责润色卡片文案

### `行为规则`

说明做事方式，例如：

- 优先忠实提取，不强行脑补
- 候选方向要彼此区分
- 语气自然、克制

### `工具约束`

说明不允许做什么，例如：

- 不改阶段推进
- 不改决策关口
- 不承担 runtime 控制职责

### `输出纪律`

说明输出格式和字段约束，例如：

- 必须输出 JSON
- 允许哪些字段
- 每个字段的数量和形态限制

### `项目规则`

项目级长期约束。

当前第一版支持通过环境变量注入：

- `PMMA_PROMPT_PROJECT`

### `追加要求`

临时追加的说明。

当前第一版支持通过环境变量注入：

- `PMMA_PROMPT_APPEND`

### `任务目标`

说明当前这次调用具体要完成什么。

## 当前优先级

当前第一版的优先级模型是：

1. 行为规则
2. 工具约束
3. 输出纪律
4. 项目规则
5. 追加要求
6. 任务目标

这还不是最终版本，但已经能避免最常见的混乱：

- 把项目级规则直接写死到任务 prompt 里
- 把临时追加要求写进身份描述里

## 当前边界

这一步并没有试图让 prompt 接管 runtime。

当前仍然明确保留在代码层的职责包括：

- 阶段推进
- 终止语义
- 执行账本
- 恢复点
- 生命周期状态

也就是说：

prompt 现在负责“怎么说、怎么约束输出”，但不负责“怎么跑 runtime”。
