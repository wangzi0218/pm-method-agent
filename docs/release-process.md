# 首次公开提交流程

## 目标

这份文档用于约束 `PM Method Agent` 第一次公开推送到 GitHub 前的最小提交流程。

它解决的不是“怎么写代码”，而是：

- 什么时候适合提交第一版
- 第一次公开前要检查什么
- 建议如何提交、打标签和写发布说明

## 适用范围

当前主要面向：

- 第一次公开推送仓库
- 第一个可对外说明的版本
- `v0.1.0` 方法内核验证版

## 建议的首次公开版本定义

建议把第一次公开版本定义为：

- 版本号：`v0.1.0`
- 阶段定位：方法内核验证版
- 主要入口：CLI
- 当前重点：稳定结构化审查卡
- 非目标：完整多轮网页交互、开放 API、复合多代理体系

## 提交前检查

首次提交前，至少确认以下内容：

### 1. 运行路径清楚

至少保证有一条外部用户可以直接跑通的路径：

- 推荐路径：`PYTHONPATH=src python3 -m pm_method_agent.cli ...`
- 可选路径：`pm-method-agent ...`

### 2. README 不依赖口头解释

README 至少应说明：

- 这个项目解决什么问题
- 它不是什么
- 当前入口是什么
- 如何运行
- 当前阶段的边界是什么

### 3. 文档与实现一致

至少检查这些文档是否与当前仓库状态一致：

- `README.md`
- `docs/output-style.md`
- `docs/evaluation-cases.md`
- `docs/release-readiness.md`

### 4. 示例可用

至少保留：

- 1 份示例案例
- 1 组可复制执行的体验命令

### 5. 测试通过

至少执行：

```bash
PYTHONPATH=src python3 -m unittest discover -s tests
```

### 6. 仓库无明显内部残留

推送前应确认：

- 没有明显只对作者自己有意义的临时说明
- 没有大面积中英混杂
- 没有明显无用文件进入仓库

## 建议的首次提交结构

第一次公开提交不建议拆得过碎。

更合理的方式是保持一个能完整表达项目当前状态的提交，例如：

```text
feat: bootstrap pm-method-agent v0.1.0
```

如果你希望更偏中文，也可以使用：

```text
feat: 初始化 pm-method-agent v0.1.0
```

## 建议的提交顺序

### 1. 本地检查

建议先执行：

```bash
PYTHONPATH=src python3 -m unittest discover -s tests
```

### 2. 查看变更范围

```bash
git status
```

### 3. 提交第一版

```bash
git add .
git commit -m "feat: bootstrap pm-method-agent v0.1.0"
```

### 4. 推送主分支

```bash
git push origin main
```

### 5. 可选：打首个标签

如果你希望把这次公开作为一个明确版本，可以继续：

```bash
git tag v0.1.0
git push origin v0.1.0
```

## 建议的首次发布说明

如果 GitHub 上要补一段 release note，建议聚焦这几件事：

### 当前已具备

- 一套面向问题定义质量的方法内核
- 一版最小可运行的 CLI
- 三个核心分析模块
- 稳定的结构化审查卡输出
- 体验用例、示例案例和最小测试

### 当前还没做

- 完整多轮会话模式
- 正式网页入口
- 开放 API
- 复合多代理运行时

### 下一步计划

- 增加多轮会话承接
- 抽出服务层
- 挂接网页与 agent 入口

## 首次公开后的节奏建议

第一次公开后，不建议马上大改产品叙事。

更合理的节奏是：

1. 先收第一轮真实体验反馈
2. 优先修正文案和阶段推进时机
3. 补评测案例
4. 再进入多轮会话和服务层开发

## 当前阶段的通过标准

如果下面这些条件同时满足，就可以进入首个 commit / push：

- CLI 主路径可运行
- README 可独立阅读
- 体验用例已经齐备
- 发布边界已经写清
- 测试通过
