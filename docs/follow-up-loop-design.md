# 连续追问闭环设计

## 目标

当前系统已经能：

- 创建案例
- 给出当前阶段卡片
- 承接用户补充
- 在部分情况下继续推进阶段

但还缺一层更明确的产品规则：

`系统到底该在什么时候追问、一次问几项、用户回答后又该怎么继续。`

这份文档只解决这一层。

## 一句话定义

连续追问闭环的目标是：

`让用户只需要自然回答，系统就能自己决定下一步是继续问、继续推进、先停下来，还是形成阶段结论。`

## 闭环最小形态

一个完整闭环应至少包含这五步：

1. 用户给出原始草稿
2. 系统输出当前判断，并指出最小必要补充
3. 用户直接回答
4. 系统把回答归并到案例状态
5. 系统决定：
   - 继续追问
   - 进入下一阶段
   - 触发决策关口
   - 暂缓或完成当前轮次

## 核心原则

### 1. 追问服务于推进，不服务于完整

不是所有缺的信息都要问。

只问：

- 会影响当前阶段判断的
- 会决定能不能进入下一阶段的
- 会改变当前结论方向的

### 2. 一次只问最小必要信息

单轮追问建议控制在：

- `1` 个问题：默认
- `2` 个问题：可以接受
- `3` 个问题：只在强相关时使用

超过 3 个，会把体验拉回问卷。

### 3. 回答后只从卡点继续

不要因为用户补了一句背景，就整轮重新理解一遍。

应优先：

- 判断回答命中了哪个卡点
- 更新相关状态
- 只从当前阻塞阶段继续

### 4. 允许用户不按顺序回答

用户经常不会正面回答系统刚问的那一句。

例如系统想问：

- 产品类型

用户却先补：

- “这个项目是诊所前台在用的 HIS。”

系统应当能从中吸收：

- ToB
- 医疗场景
- 前台角色

而不是机械地再问一遍。

## 哪些情况应该追问

### 情况 1：缺最小场景信息

例如缺：

- 产品类型
- 主要平台
- 关键角色

这时追问优先级最高。

### 情况 2：存在多个都说得通的理解方向

例如一句话里同时混着：

- 现象
- 解释
- 方案

或同时存在：

- 多个问题定义方向

这时要先追问，不应直接收敛到单一路径。

### 情况 3：已经到决策关口，但还缺关键依据

例如已经在判断“值不值得做”，但缺：

- 时机依据
- 机会成本
- 非产品路径是否已试过

这时可以继续追问 1 到 2 项，再决定是否真的阻塞。

## 哪些情况不该继续追问

### 情况 1：当前已足够形成阶段结论

例如已经能判断：

- 输入偏方案导向
- 现状证据明显不足

这时应直接给卡片，而不是为了“更完整”继续问。

### 情况 2：已经到人工决策点

例如当前真正缺的不是信息，而是：

- 你到底要不要继续产品化

这时要让用户做选择，而不是继续追问外围信息。

### 情况 3：用户明显表达了暂缓倾向

例如：

- “先不做”
- “资源不够”
- “先观察”

这时优先进入 `deferred` 或决策关口，而不是再补更多分析。

## 追问的优先级顺序

建议统一按这个顺序选择问题：

1. 能决定阶段推进的
2. 能明显缩小问题定义空间的
3. 能显著改变价值判断的
4. 能决定验证动作是否成立的
5. 其他补充性信息

## 单轮后的判定逻辑

每轮回答后，系统都应该先做一次轻判定：

### A. 这轮回答属于哪类输入

- 直接回答刚才的问题
- 补充新的背景
- 更正前面的理解
- 表达决策倾向
- 只是问“下一步该做什么”

### B. 这轮回答命中了哪些卡点

- 场景信息
- 问题定义
- 决策依据
- 验证信息
- 人工选择

### C. 这轮之后应该进入哪种结果

- 继续追问
- 推进阶段
- 触发关口
- 暂缓
- 完成当前轮次

## 推荐的终止语义

连续追问闭环里，至少要稳定区分：

- `continued`
  含义：这轮已经接住，但还没到最终结论

- `blocked`
  含义：必须先补信息或做选择

- `deferred`
  含义：当前建议先暂缓

- `completed`
  含义：当前阶段已形成可交付卡片

不要把这些都混成“完成”。

## 工作区与记忆如何参与

连续追问闭环并不只依赖当前一句回答。

它至少要读取三层上下文：

### 1. 当前案例上下文

用于判断：

- 现在卡在哪
- 刚才问过什么
- 已经补上什么

### 2. 项目背景

用于判断：

- 默认产品类型
- 默认平台
- 长期约束

### 3. 长期偏好

这一层暂时只建议保留最小能力，例如：

- 输出更偏简洁还是偏展开
- 是否更关注 ToB / ToC 视角
- 默认更重视哪些指标

## 产品层最小实现建议

`v0.2` 不需要一口气把所有复杂交互都做完。

建议先实现下面这条最小链路：

1. 原始输入进入
2. 系统给第一张卡
3. 系统同时给出 `1-3` 个最小必要追问
4. 用户自然语言回答
5. 系统更新案例状态
6. 系统明确输出：
   - 现在进入哪一阶段
   - 还差什么
   - 下一步最建议补什么

## 对代码层的影响

下一步代码层更适合补这些点：

### 1. 追问选择器

负责从当前案例状态里选出：

- 这一轮最该问的 1 到 3 项

### 2. 回答归并器

负责把用户这句回答归并进：

- 场景信息
- 证据
- 决策倾向
- 角色关系

### 3. 推进判定器

负责决定：

- 继续追问
- 进入下一阶段
- 转关口
- 暂缓

### 4. 外壳提示统一化

CLI / agent / 网页都应能一致展示：

- 这轮在补什么
- 为什么要补
- 补完后会去哪

## 回归样本建议

为了避免连续追问越做越像脚本，建议后面补一组专门样本：

- 用户正面回答
- 用户答非所问
- 用户补的是新的背景
- 用户在同一句里既回答又纠正前提
- 用户表达暂缓
- 用户表达“先继续做产品化”

## 这一层做完的标志

如果连续追问闭环做得对，应该出现这些变化：

- 用户不用自己重写整段上下文
- 用户能明显感觉系统是在“继续同一个判断”
- 追问轮次不多，但每轮都在推进
- 该停的时候真的会停

做到这里，`PM Method Agent` 才真正开始从“会分析”走向“会协作”。
