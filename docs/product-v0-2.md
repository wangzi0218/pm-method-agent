# v0.2 产品定义

## 这版要解决什么

`PM Method Agent` 到现在已经有：

- 方法主线
- 多轮工作区与案例
- 统一 agent 入口
- 本地 HTTP 服务
- 最小网页 demo
- 最小 runtime 可观察能力

但从产品角度看，当前还有一个明显缺口：

`用户可以开始分析，但还不容易把一个模糊草稿稳定推进到“可判断、可暂缓、可继续验证”的状态。`

也就是说，现在更像：

- 一个能跑通方法链路的系统

还不像：

- 一个能连续接住产品经理思路、主动追问、减少来回重写的协作产品

## 一句话目标

`v0.2` 的目标不是继续扩更多分析模块，而是：

`把当前系统升级成一个能主动追问 2 到 4 轮、稳定推进问题判断的协作产品。`

## 目标用户

优先面向这三类人：

- 中初级产品经理：有问题草稿，但不一定知道下一步该补什么
- 独立负责需求判断的人：需要快速判断“值不值得做”
- 已经习惯和 AI 协作的人：但不希望每次都自己手写结构化 prompt

## 核心使用场景

### 场景 1：模糊草稿进入

用户只给一句还没想透的话，例如：

- “前台最近老是漏提醒患者，我在想是不是要处理一下。”
- “新用户发帖率还是起不来，我没想清楚问题到底卡在哪。”

系统目标：

- 不要求用户先整理成完整需求
- 能先输出一个可读的判断
- 能明确追问最小必要信息

### 场景 2：连续补充推进

用户按系统提示继续补：

- 场景背景
- 失败案例
- 角色关系
- 证据
- 决策倾向

系统目标：

- 只从当前卡点继续
- 不要每轮都像重新开一个新问题
- 能自然进入下一阶段或下一关口

### 场景 3：形成阶段结论

系统最终要把一段模糊对话推进到以下三种结果之一：

- 现在可以继续往方案或验证走
- 现在更适合先补证据或补背景
- 现在更适合暂缓，不进入产品化

## v0.2 的核心交付

这一版真正要交付的不是“又多一个入口”，而是下面四件事：

### 1. 连续追问闭环

用户不需要自己判断下一轮该怎么问。

系统应当能：

- 判断当前是否该追问
- 一次只问 1 到 3 个最影响推进的问题
- 根据回答继续推进，而不是整轮重跑

### 2. 稳定的阶段推进

系统需要更稳定地区分：

- 还没到问题定义
- 已经能做问题定义，但还没到决策挑战
- 已经到决策挑战，但需要先做选择
- 已经可以进入验证设计

### 3. 记忆分层对用户可见

当前已经有技术层的工作区、案例和 runtime 记忆。

`v0.2` 需要把它变成用户可理解的产品能力：

- 哪些信息属于当前案例
- 哪些属于项目背景
- 哪些是长期偏好

### 4. 统一入口体验

`CLI / agent / 网页 demo` 至少要在产品行为上保持一致：

- 追问逻辑一致
- 阶段推进一致
- 阻塞提示一致
- 历史承接一致

## 这版不做什么

为了控制范围，`v0.2` 暂时不把这些作为主目标：

- 不做复合 agent 编排
- 不做并行专项智能体委派
- 不做真正的在线托管服务
- 不做复杂团队协作或多人共享工作台
- 不做完整 MCP 外壳
- 不把网页 demo 做成完整后台

这些不是不做，而是放到后面。

## 成功标准

如果 `v0.2` 做得对，至少应该满足：

### 用户体验

- 用户带一句模糊草稿进入，系统能接住
- 用户连续补 2 到 4 轮，不需要自己重写完整上下文
- 用户能明确知道现在卡在哪、为什么卡、补完后会去哪

### 方法质量

- 系统不会因为多轮补充而乱跳阶段
- 系统不会为了“信息完整”而过度追问
- 系统在该停的时候会明确停下来让人做决策

### 产品一致性

- CLI、网页和 agent 入口对同一段对话给出相近推进结果
- 阻塞、暂缓、继续、完成这些终止语义在不同入口一致

## 衡量指标

建议先用这几类指标判断 `v0.2` 是否站得住：

### 体验指标

- 从首句输入到形成明确阶段结论的平均轮次
- 用户在 3 轮内补齐关键场景信息的比例
- 用户重复重写同一问题的比例

### 质量指标

- 阶段误判样本数
- 明显过问或漏问样本数
- 用户在阻塞阶段仍被错误推进的样本数

### 产品指标

- 网页 demo / agent 入口的连续使用完成率
- 示例装载后的真实试用转化率
- 真实问题试跑后“值得继续用”的主观反馈

### 已有基础

- 已有最小工作区和案例状态
- 已有回复解释与前置收敛
- 已有最小阶段推进和关口控制
- 已有 runtime session、恢复点和压缩记忆
- 已有网页 demo 与统一入口

### 主要缺口

- 还缺一套更清晰的“连续追问产品规则”
- 还缺“什么时候问、问几项、什么时候停”的明确策略
- 还缺用户能直接感知的背景记忆和长期偏好心智
- 还缺面向 `v0.2` 的产品级回归样本

## 建议的执行顺序

`v0.2` 更适合按这个顺序推进：

1. 先定产品目标和非目标
2. 再定义连续追问闭环
3. 再补工作区记忆与项目背景的产品层表达
4. 再做最小实现与回归样本
5. 最后再决定是否进入更复杂的复合 agent

## 这一版做完后，项目会变成什么

如果 `v0.2` 做完，`PM Method Agent` 的定位会更清楚：

- 不只是一个“会产出审查卡”的系统
- 而是一个“能把模糊草稿连续推进到判断结果”的协作产品

这会是从“可运行 demo”到“真正可持续使用”的第一次关键跨越。
