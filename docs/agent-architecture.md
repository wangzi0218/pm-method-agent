# Agent 架构

## 目标

`PM Method Agent` 在产品形态上应当优先按 `agent` 来设计，而不是把命令行工具直接暴露给最终用户。

原因是：

- 用户更可能在 IDE、AI 工具或聊天窗口中调用它
- 复杂场景需要分阶段判断、追问和协作
- 用户需要的是一个统一入口，而不是一组底层模块或命令

因此，这个项目应当采用：

- `用户入口 = 主代理`
- `内部执行 = 编排器 + 分析模块 / 专项代理`
- `命令行工具 = 开发、验证、回归入口`

这并不意味着未来只能以 `agent` 交付。

更准确的理解应当是：

- `agent` 是方法入口的优先抽象
- `CLI` 是当前阶段的验证入口
- `网页` 是未来面向更广泛用户的可视化入口

三者应共用同一套状态机、服务层和输出契约。

## 总体结构

建议采用三层结构：

### 1. 主代理

用户唯一看到的入口。

主代理负责：

- 接收用户输入
- 判断是否需要先补场景基础信息
- 决定进入哪个分析阶段
- 控制人工决策点
- 汇总并输出审查卡

### 2. 方法协调层

方法协调层由 `orchestrator`、`case state`、契约和决策关口组成。

它负责：

- 维护统一状态
- 调用内部能力
- 合并结果
- 控制阶段推进

### 3. 专项能力层

专项能力层可以分阶段演进：

- 第一阶段：内部分析模块
- 第二阶段：半独立专项代理
- 第三阶段：可协作的多代理体系

## 为什么不是一开始就做全多代理

虽然最终产品入口应当是 `agent`，但内部一开始不应直接做成完全自治的多代理系统。

原因是：

- 复杂度高
- 输出容易冲突
- 评测和回归更难
- 提示词维护成本更高

更稳的路径是：

1. 先把主代理定义清楚
2. 内部先用结构化分析模块支撑
3. 再把高价值模块逐步升级成专项代理

## 主代理的职责边界

主代理必须保留以下职责，不建议全部下放：

- 用户意图识别
- 场景信息是否足够的判断
- 当前阶段推进判断
- 是否触发人工决策关口
- 最终输出的审查卡整合

换句话说，主代理是“最终负责的人”，不是纯转发器。

## 什么时候触发专项代理

以下场景更适合触发专项代理或专项分析模块：

- 用户输入高度模糊，需要先做问题定义拆解
- 场景复杂，存在多类利益冲突
- 明显需要挑战“是否值得做”
- 已接近立项，需要明确最小验证动作和成功标准

建议的专项能力拆分：

- `context-alignment-agent`
- `problem-framing-agent`
- `decision-challenge-agent`
- `validation-design-agent`

第一版不一定真的都实现成独立代理，但职责应先按这个方式定义。

## 一次典型调用的流程

### 步骤 1：用户输入

例如：

“前台希望增加一个预约前提醒弹窗，避免漏提醒患者。”

### 步骤 2：主代理判断是否缺场景信息

如果缺少关键场景信息，先进入“场景对齐”：

- 这是企业产品、消费者产品还是内部产品
- 主要平台是什么
- 谁提需求、谁使用、谁承担结果

### 步骤 3：主代理调用问题定义能力

输出：

- 输入是否偏方案导向
- 当前问题是否定义清楚
- 关键角色是否明确

### 步骤 4：主代理判断是否进入决策挑战

如果问题定义已具备基本条件，则进入下一阶段。

### 步骤 5：主代理调用决策挑战能力

输出：

- 是否有明确时机
- 是否应优先尝试非产品路径
- 是否值得投入产品能力

### 步骤 6：必要时调用验证设计能力

输出：

- 可证伪假设
- 最小验证动作
- 成功指标和停止条件

### 步骤 7：主代理输出审查卡

统一输出：

- 基础信息
- 初步判断
- 审查卡片
- 决策关口
- 建议动作
- 待补信息

## 命令行工具的定位

命令行工具不是最终用户入口，而是：

- 开发调试入口
- 测试与回归入口
- 示例演示入口

它的价值是帮助我们验证方法内核，而不是要求普通用户用命令行完成日常分析。

## 近期建议

下一阶段的工作重点应当是：

1. 把主代理的用户交互定义清楚
2. 明确哪些问题由主代理直接判断，哪些问题交给专项能力
3. 保持 `case state` 和审查卡为统一中间层
4. 在不破坏当前 Python 内核的前提下，增加 agent 入口适配层
