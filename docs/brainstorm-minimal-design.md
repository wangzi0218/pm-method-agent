# Brainstorm 最小实现方案

## 文档目标

这份文档不再讨论“要不要做 brainstorm”，而是明确一件事：

`如果要做，它应该怎样并入 PM Method Agent 的主线，而不是长成另一套平行流程。`

本文聚焦四个问题：

- 它在主代理里到底扮演什么角色
- 什么时候触发，什么时候旁路
- 它产出什么中间结果，怎么服务后续判断
- 第一版代码应该落到哪里，做到什么程度算够

## 主线定位

`brainstorm` 在这个项目里不应是独立入口，也不应是一条完整平行流程。

更合适的定义是：

- 主代理内部的一层前置收敛能力
- 位于用户原始输入和正式问题定义之间
- 只在高不确定性输入下短暂出现
- 目标是帮助主代理先看清方向，再继续推进

换句话说，它服务的是主线，不是替代主线。

主线仍然是：

- 场景对齐
- 问题定义
- 决策挑战
- 验证设计

`brainstorm` 只是帮助主线少走弯路，不负责单独给结论。

## 要解决的主线问题

把这一层接进来，主要是为了修正当前主线里的两个体验问题：

### 1. 过早收敛

当用户只给一句模糊草稿时，系统容易太快进入“判断模式”，让体验看起来像流程化机器人。

### 2. 模糊输入缺少过渡层

很多真实输入会同时混着：

- 现象
- 解释
- 方案倾向
- 角色关系
- 紧迫性感受

如果没有一个很轻的过渡层，主代理就容易把一句话过早当成明确问题。

## 不做什么

为了保证它不偏离主线，第一版需要明确几条边界。

- 不做独立的 `brainstorm agent`
- 不增加新的用户心智入口
- 不做无限发散式陪聊
- 不把候选理解直接当成正式结论
- 不在后续阶段自动反复回退到这一层

## 在状态机里的位置

建议把这一层命名为 `pre_framing`，而不是直接叫 `brainstorm`。

这样更符合项目主线，也更适合后续扩展成：

- 启发式版前置收敛
- LLM 增强版前置收敛
- 其他更细的理解辅助动作

推荐状态关系如下：

```text
intake
  -> pre_framing?
  -> context_alignment
  -> problem_definition
  -> decision_challenge
  -> validation_design
  -> done / blocked / deferred
```

这里的关键约束是：

- `pre_framing` 只发生在主线前段
- 它只影响“先问什么、先看哪条理解”
- 它不改变后续阶段职责

## 触发原则

### 适合触发的情况

满足以下任一类信号时，可以进入 `pre_framing`：

- 输入明显模糊，只能看出大致困扰，不能稳定定位问题
- 输入同时混着问题、解释和方案
- 同一句话存在 2 到 3 条都说得通的理解路径
- 用户显式要求“一起想想”“发散一下”“还有没有别的理解”

### 不适合触发的情况

以下情况应直接旁路：

- 场景信息已经足够清楚
- 当前输入已经接近事实型问题描述
- 用户明确要求直接判断是否值得做
- 当前案例已经进入 `decision_challenge` 之后的阶段

### 触发阈值建议

第一版不要追求复杂评分模型，可以先用启发式组合判断：

- 模糊表达命中
- 方案表达命中
- 多角色或多目标冲突命中
- 缺少明确现象与证据

只要达到阈值，就先走一张轻量继续卡，而不是立刻输出完整审查卡。

## 中间产物契约

这一层必须有结构化中间产物，不能只靠文案。

建议新增 `pre_framing_result`：

```json
{
  "triggered": true,
  "reason": "输入同时混有现象和方案倾向，且缺少稳定问题边界。",
  "candidate_directions": [
    {
      "direction_id": "D-001",
      "label": "提醒流程执行不稳定",
      "summary": "当前更像人工流程和责任链不稳定，而不是单点功能缺失。",
      "assumptions": [
        "关键动作依赖人工记忆",
        "提醒责任没有稳定落到某个角色"
      ],
      "confidence": "medium"
    },
    {
      "direction_id": "D-002",
      "label": "复诊患者识别或触发条件不稳定",
      "summary": "问题可能不在提醒动作本身，而在该提醒谁、何时提醒的规则。",
      "assumptions": [
        "系统里缺少稳定的患者分层或触发条件",
        "前台和店长对漏提醒的判定标准不一致"
      ],
      "confidence": "low"
    }
  ],
  "priority_questions": [
    "漏提醒主要发生在哪个环节",
    "当前是谁在日常流程里负责提醒",
    "最近为什么开始觉得这件事更重要"
  ],
  "recommended_direction_id": "D-001"
}
```

这份结果的定位要非常克制：

- 后续模块可以参考
- 但不能把它当成已证实事实
- 用户可以直接修正其中任何方向

## 用户可见输出

对用户不直接暴露 `pre_framing_result` 这个名字。

用户看到的仍然应该是一张贴近主线的轻卡，而不是“brainstorm 卡”。

推荐名称：

- `继续卡`
- `先收一收`
- `先看哪条更像`

第一版更建议统一为：

- `继续卡`

## 继续卡的最小槽位

继续卡不宜太重，建议固定为五块：

### 1. 当前判断

一句话告诉用户为什么没有直接进入完整审查。

例如：

- 这件事现在还有几种说得通的理解，先别急着往方案走。

### 2. 更像哪几类问题

只保留 2 到 3 个方向，不要列太多。

### 3. 先确认这几件事

只问最影响后续推进的 2 到 3 个问题。

### 4. 更建议先沿哪条继续

这一项必须有，避免只发散不收口。

### 5. 补完后进入哪里

明确告诉用户接下来是：

- 回到场景对齐
- 进入问题定义
- 继续当前阶段

## 和现有输出体系的关系

这一层不是新开一种产品形态，而是补足现有输出体系。

建议形成以下关系：

- 场景信息缺失明显：输出 `场景补充卡`
- 需要先拉开理解：输出 `继续卡`
- 当前阶段被决策挡住：输出 `决策关口卡`
- 当前阶段已能稳定判断：输出 `完整审查卡`

这意味着：

- `继续卡` 是主线前段的轻收敛卡
- 它和 `场景补充卡` 并列，但更偏理解方向，而不是信息采集

## 和场景对齐的衔接

`pre_framing` 不应替代场景对齐。

更自然的关系是：

- 如果当前既缺场景信息，又存在多种理解路径
- 系统也先用 `继续卡` 指出“更像哪几类问题”
- 但尾部仍要把用户带回“先补场景信息”

也就是说，`pre_framing` 可以决定“先补什么”，
但最终仍由 `context_alignment` 承接最小场景要素补齐。

## 和问题定义的衔接

`problem_definition` 可以读 `pre_framing_result`，但只能把它作为弱提示。

建议遵守三条规则：

- 不把候选方向写成已确认事实
- 不因为推荐方向存在，就忽略用户后续修正
- 只有当用户补充内容与候选方向持续一致时，才逐步提高判断信心

## 和 LLM 适配层的关系

这层设计必须兼容两种实现：

### 1. 启发式实现

适合第一版。

特点：

- 不依赖模型
- 成本低
- 易回归
- 先验证“是否能改善过早收敛”

### 2. LLM 增强实现

放在后续版本。

更适合补这些能力：

- 候选方向生成更自然
- 方向排序更稳
- 继续卡文案更像真实协作对话

LLM 接入仍应走现有 `LLMAdapter`，并保持 OpenAI-compatible 配置方式。

### 最小实现建议

第一版只需要做到：

1. 判断当前输入是否要进入 `pre_framing`
2. 生成 2 到 3 个候选方向
3. 生成 2 到 3 个优先确认问题
4. 渲染一张继续卡
5. 把中间产物挂回 case state

### 暂时不用做的事

第一版可以先不做：

- 候选方向的多轮排序
- 用户显式选择方向后的复杂分支
- 基于历史偏好的个性化发散
- 复杂的 LLM prompt 编排

## 回归检查

把这一层接进主线后，至少要新增三类检查：

### 1. 不过早收敛

对模糊输入，系统不应直接给完整审查卡。

### 2. 不发散过度

候选方向不应超过 3 个，问题不应超过 3 个。

### 3. 继续路径明确

继续卡必须明确：

- 更建议从哪里补
- 补完后进入哪个阶段

## 何时可以继续往代码落

满足以下条件时，就可以进入实现：

- 主线角色已经清楚：它是 `pre_framing`，不是平行产品
- 输出契约已经确定：有结构化中间产物，也有轻卡形态
- 和状态机的关系已经清楚：只插在主线前段
- 启发式第一版边界已经收紧：有限发散，立即收口

## 一句话总结

`brainstorm` 在 PM Method Agent 里最合适的落点，不是独立使用，而是作为主代理内部的一层 `pre_framing` 能力，帮助系统在模糊输入面前先看清方向，再把用户带回主线继续推进。
