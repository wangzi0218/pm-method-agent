# 主代理交互设计

## 文档目标

本文档定义 `PM Method Agent` 的主代理交互方式，用来回答三个问题：

1. 用户如何触发和使用主代理
2. 主代理如何决定下一步做什么
3. 代码层需要支持哪些运行效果

这份文档不讨论“某个分析模块内部怎么推理”，而是聚焦主代理的交互状态机。

## 一句话定位

`PM Method Agent` 的主代理应该像一个“会推进需求审查流程的协作代理”，而不是一次性吐出完整分析结果的脚本。

它的职责不是把所有内容一次说完，而是：

- 判断当前信息够不够
- 在必要时追问最少的问题
- 控制阶段推进
- 在合适的节点输出审查卡

## 用户触发方式

主代理适合通过统一入口触发，例如：

- “帮我审查这个需求”
- “帮我做问题定义审查”
- “帮我判断这个需求值不值得做”
- “帮我梳理这个需求现在该不该推进”

从交互体验上，用户不应主动选择内部模块；主代理应根据输入自动决定进入哪个阶段。

## 主代理的核心职责

主代理应始终保留以下职责：

- 判断当前输入是否足以开始分析
- 判断是否缺少场景基础信息
- 决定进入问题定义、决策挑战还是验证设计
- 判断是否需要人工决策
- 汇总专项能力结果并输出统一审查卡

以下职责不应完全交给专项能力：

- 阶段推进
- 决策关口触发
- 用户追问顺序控制
- 最终输出整合

## 主代理状态机

建议把主代理设计为以下状态：

### 1. `intake`

含义：
接收用户原始输入，形成初始案例状态。

主代理在这一状态需要完成：

- 接收原始问题描述
- 判断输入是否过短、过空泛、过于偏方案
- 判断是否已有足够的场景基础信息

可能流转到：

- `pre_framing`
- `context_alignment`
- `problem_definition`

### 2. `pre_framing`

含义：
在正式问题定义前，先对模糊输入做一次有限收敛。

只在以下情况下进入：

- 输入同时混着问题、解释和方案倾向
- 同一句话存在多条都说得通的理解路径
- 直接进入判断容易过早收敛

主代理在这一状态需要完成：

- 生成 2 到 3 个候选理解方向
- 指出最值得先确认的 2 到 3 个问题
- 明确更建议先沿哪条继续

可能流转到：

- `context_alignment`
- `problem_definition`

### 3. `context_alignment`

含义：
补齐最小场景基础信息。

只在以下情况下进入：

- 缺少产品类型
- 缺少主要平台
- 缺少关键用户角色
- 缺少会显著影响判断的强约束

主代理在这一状态需要完成：

- 决定要问哪些问题
- 一次最多追问 1 到 3 个高价值问题
- 把补充结果写回案例状态

可能流转到：

- `problem_definition`
- `blocked`

### 4. `problem_definition`

含义：
判断当前输入是否是一个足够明确的问题，而不是方案表达。

主代理在这一状态需要完成：

- 调用问题定义能力
- 判断是否存在明显的方案导向
- 判断关键角色是否已初步识别
- 判断是否具备进入下一阶段的最低条件

可能流转到：

- `decision_challenge`
- `blocked`

### 5. `decision_challenge`

含义：
判断这个问题是否值得在当前阶段推进，以及是否应优先产品化。

主代理在这一状态需要完成：

- 调用决策挑战能力
- 判断是否存在时机依据
- 判断是否应优先评估非产品路径
- 判断是否需要用户做阶段决策

可能流转到：

- `validation_design`
- `blocked`
- `deferred`

### 6. `validation_design`

含义：
在准备继续推进时，定义最小验证路径。

主代理在这一状态需要完成：

- 调用验证设计能力
- 输出可证伪假设
- 输出最小验证动作
- 输出成功指标、护栏指标和停止条件

可能流转到：

- `done`
- `blocked`

### 7. `blocked`

含义：
当前不能继续推进，必须先补信息或做决策。

这一状态下主代理必须明确输出：

- 为什么被阻塞
- 缺什么
- 谁来补
- 补完之后进入哪里

### 8. `done`

含义：
当前轮次已经形成完整审查卡，可以交给用户继续推进下一步工作。

### 9. `deferred`

含义：
根据当前信息，建议暂缓，不进入后续验证或方案阶段。

## 追问规则

主代理的追问必须满足“少而准”。

### 追问原则

- 一次只问最影响判断的内容
- 优先问场景，不优先问细节
- 优先问能决定阶段推进的问题
- 不要为了“信息完整”而追问

### 推荐追问顺序

1. 产品类型：企业产品、消费者产品还是内部产品
2. 主要平台：桌面端、移动端、小程序还是多端
3. 关键用户角色：谁提、谁用、谁承担结果
4. 关键约束：合规、设备能力、交付周期等

### 不该追问的情况

如果缺失信息不会影响当前阶段判断，就不追问。

例如：

- 已经足够判断“当前输入偏方案导向”
- 已经足够判断“现状证据不足”

这类情况下，主代理应直接输出审查卡，而不是继续问细节。

## 前置收敛规则

`pre_framing` 不是独立流程，而是主线前段的可选动作。

建议遵守以下规则：

- 只在高不确定性输入下触发
- 一次只给 2 到 3 个候选方向
- 一次只问 2 到 3 个最关键的问题
- 必须给出更建议先往哪条继续
- 不把候选方向直接当成正式结论

如果用户输入已经足够清楚，就直接旁路这一层。

## 阶段推进规则

### 进入 `problem_definition` 的条件

- 有原始问题描述
- 场景基础信息即使不完整，也不妨碍做第一轮问题定义判断

### 进入 `pre_framing` 的条件

- 输入模糊到不足以稳定落入单一理解
- 存在明显的方案先行表达
- 当前更适合先拉开问题空间，再决定补什么

### 从 `pre_framing` 进入 `context_alignment` 的条件

- 当前仍缺关键场景信息
- 已经能看出先该补哪类场景要素

### 从 `pre_framing` 进入 `problem_definition` 的条件

- 候选方向已经足够收敛
- 即使场景信息不完整，也不妨碍做第一轮问题定义判断

### 从 `problem_definition` 进入 `decision_challenge` 的条件

- 已经能初步区分问题与方案
- 关键角色已有基本判断
- 当前输入不再只是空泛线索

### 从 `decision_challenge` 进入 `validation_design` 的条件

- 没有被明确建议暂缓
- 不存在明显更优的非产品路径尚未评估
- 已经具备继续验证的基本价值判断

### 进入 `blocked` 的条件

- 缺少关键场景信息，无法继续判断
- 当前问题定义过于空泛
- 当前结论依赖用户做明确选择

## 决策关口触发规则

主代理不应把所有不确定项都做成决策关口。

只有以下情况才生成决策关口：

- 是否已可以进入方案设计阶段
- 是否应优先评估非产品路径
- 是否值得在当前阶段继续投入
- 是否建议暂缓

一个决策关口至少要包含：

- 问题
- 可选项
- 默认建议
- 原因
- 是否阻塞当前流程

## 默认输出类型

主代理应支持四类用户可见输出：

### 1. 场景补充卡

用于：

- 当前缺少关键场景基础信息

应包含：

- 当前缺什么
- 为什么要补
- 建议用户回答的问题

### 2. 继续卡

用于：

- 当前输入存在多种合理理解
- 系统需要先做一次有限收敛

应包含：

- 当前为什么先不直接下判断
- 当前更像哪几类问题
- 最值得先确认的几件事
- 更建议先沿哪条继续

### 3. 阶段阻塞卡

用于：

- 当前阶段不能继续推进

应包含：

- 阻塞原因
- 需要补什么
- 补完后进入哪个阶段

### 4. 决策关口卡

用于：

- 当前需要用户做明确决策

应包含：

- 当前问题
- 选项
- 推荐项
- 理由

### 5. 完整审查卡

用于：

- 当前可以输出完整阶段性分析结果

应包含：

- 基础信息
- 当前判断
- 关键判断
- 决策关口
- 建议先做
- 建议补充

## 对代码层的要求

文档定完后，代码层不需要立刻实现真正的多代理，但至少应达到以下效果。

### 1. 支持主代理状态流

当前运行时应从“线性跑三个模块”升级为“按状态推进”。

至少要支持：

- `intake`
- `pre_framing`
- `context_alignment`
- `problem_definition`
- `decision_challenge`
- `validation_design`
- `blocked`
- `done`
- `deferred`

这些状态建议直接体现在 `case state` 中，而不是只留在运行时内部。

### 2. 支持可中断推进

运行时不能默认把所有阶段都跑完。

要支持：

- 在场景信息不足时停下
- 在问题定义不成立时停下
- 在需要人工决策时停下
- 只在条件满足时进入下一阶段

### 3. 支持标准化追问

运行时应能表示：

- 当前缺哪些场景信息
- 当前建议追问哪几个问题
- 当前等待用户补什么

建议落到统一字段：

- `blocking_reason`
- `pending_questions`
- `output_kind=context-question-card`

### 4. 支持主代理视角输出

输出层除了展示分析结果，还要展示：

- 当前停在哪个阶段
- 为什么停在这里
- 现在最需要用户补什么

建议至少支持：

- `context-question-card`
- `continue-guidance-card`
- `stage-block-card`
- `decision-gate-card`
- `review-card`

### 5. 支持专项能力调用接口

即使当前内部还是 Python 模块，也应当统一通过接口调用：

- 问题定义能力
- 决策挑战能力
- 验证设计能力

这样后续把某些能力升级成专项代理时，主流程不需要重写。

## 近期实现建议

建议按下面顺序推进：

1. 先在代码里实现主代理状态机
2. 再实现“场景补充卡 / 继续卡 / 阶段阻塞卡 / 决策关口卡”
3. 再把内部模块调用统一收口
4. 最后再决定哪些模块要升级成专项代理
