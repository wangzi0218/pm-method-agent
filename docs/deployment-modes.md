# 部署形态

## 目标

`PM Method Agent` 不应该被绑定成单一形态。

它更适合被拆成两层来看：

- 上层是用户可见入口
- 下层是共用的方法内核、状态机、规则层和运行时

这样 CLI、IDE、网页、云端服务才不会各做各的。

## 一句话结论

当前最合理的方向不是“只做本地”或“只做云端”，而是：

- 先用本地形态把运行时站稳
- 再逐步补网页 / 云端入口
- 长期优先支持混合模式

也就是：

- 分析主线可以在云端或托管服务运行
- 本地环境能力通过受控工具层接入

## 三种主要形态

### 1. 本地形态

典型入口：

- CLI
- IDE agent
- 本地 skill
- 本地 HTTP service

典型特点：

- 最容易接本地文件、目录、命令和仓库规则
- 最适合验证 runtime policy、hook、tool ledger 这类底座能力
- 最适合深度协作和开发场景

适合的用户：

- 在 IDE 里持续和 agent 协作的人
- 需要读写本地工程的人
- 需要把仓库规则、目录规则直接接进运行时的人

当前仓库已经覆盖的主要是这一层。
不过现在已经开始把“本地工具”和“平台工具”放进同一套发现接口里，为后面的混合模式先打边界。

### 2. 云端 / 网页形态

典型入口：

- 托管网页
- SaaS 服务
- 团队共享工作台
- 对外 API

典型特点：

- 用户上手成本更低
- 更容易做团队级共享、沉淀和协作
- 更适合不依赖本地环境的产品分析场景

但这类形态天然有边界：

- 不能直接假设能访问用户本地文件
- 不能直接假设能执行用户机器上的命令
- 需要更严格地区分“平台内工具”和“用户本地工具”

所以网页不是不能做，而是不能把“本地执行”当默认前提。

### 3. 混合形态

这是长期最值得做的形态。

典型结构：

- 云端主控负责会话、状态机、规则、记忆和结果组织
- 本地轻执行器负责受控访问本地环境

这种方式的好处是：

- 网页和托管服务可以统一用户体验
- 本地能力仍然能保留
- 可以把敏感动作收敛到更小的本地执行面
- 更适合未来接 MCP、IDE 插件或桌面端壳

## 为什么现在先做本地底座

不是因为产品只能本地化，而是因为本地最容易先验证这些核心问题：

- query loop 怎么跑
- 跨轮状态怎么恢复
- 工具调用怎么留账
- 阻断、失败、继续、恢复这些终止语义怎么区分
- 规则怎么作用到真实读写和命令执行

这些问题如果不先收敛，直接做网页，只是把复杂度往后拖。

## 不同形态下，哪些层应该共用

无论最后是本地、网页还是混合，下面这些层都应该共用：

- 方法内核
- 案例状态与工作区状态
- 决策关口
- 输出契约
- prompt layering
- rule loading
- runtime policy
- operation enforcement
- runtime session / execution ledger

真正应该因形态而变化的，主要是：

- 入口层
- 工具适配层
- 鉴权与部署方式

## 工具层如何按形态拆分

### 本地工具

典型包括：

- 读取本地文件
- 写入本地文件
- 列目录
- 搜索文本
- 执行受控命令

这些工具更适合由本地执行器承接。

### 平台工具

典型包括：

- 读取案例
- 更新项目背景
- 查询工作区
- 调用模型
- 调用平台内知识库

这些工具更适合由云端或服务端承接。

当前仓库已经开始补这层最小实现，例如：

- `platform-workspace-overview`
- `platform-case-read`
- `platform-project-profile-read`
- `platform-project-profile-upsert`

其中 `platform-project-profile-upsert` 这一类可写平台工具，已经会经过统一 runtime、hook 和动作级策略，而不是绕开运行时直接改状态。

### 形态设计原则

不要把“是否在本地执行”写死在 agent 主线里。

更合理的方式是：

- agent 只声明自己需要什么能力
- runtime 决定这个能力由本地工具还是平台工具承接

## 当前推荐路径

### 近期

- 继续把本地运行时补稳
- 补更多受控本地工具
- 保持 HTTP service 作为统一适配层

### 中期

- 补网页壳
- 把网页入口接到同一套会话服务
- 区分平台工具和本地工具

### 长期

- 形成混合架构
- 云端主控 + 本地轻执行器
- 支持 IDE、网页、API、MCP 多入口共用同一套 runtime

## 对 README 的影响

README 更适合保留这些内容：

- 项目定位
- 适合谁 / 不适合谁
- 一个最小例子
- 最短可运行方式
- 文档导航

而不适合继续堆这些内容：

- 大量接口细节
- 所有接入方式的完整示例
- 过长的阶段状态清单
- 过深的架构分支解释

这些内容更适合下沉到独立文档。

## 当前判断

所以结论很明确：

- 项目不需要被定义成“本地 agent”
- 也不应该过早只押注网页
- 当前先补本地底座是合理的
- README 现在确实应该继续解耦，保留首页角色，把部署形态和接入细节拆出去
