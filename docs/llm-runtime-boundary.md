# LLM 与 Runtime 能力边界

## 文档目标

这份文档回答的是一个很实际的问题：

`PM Method Agent` 后面继续增强时，哪些能力应该交给 LLM，哪些能力必须留在 runtime，哪些能力适合走“runtime 主判 + LLM 辅助”的混合路径。

如果这条边界不先收住，项目很容易走向两个极端：

- 一有语义问题就往 prompt 里加，最后系统越来越像“会说话的工作流”
- 明明已经进入长尾语义问题，还继续用规则硬撑，体验越来越像“程序化机器人”

这两种方向都不适合这个项目。

## 一句话原则

更合理的分工是：

- runtime 负责稳定主线、状态真相和硬约束
- LLM 负责开放语义理解、表达增强和长尾补偿
- 混合能力由 runtime 定义输入输出契约，再让 LLM 在契约内工作

一句话概括：

`LLM 负责理解和生成，runtime 负责约束、记账和收口。`

## 为什么不能把一切都交给 LLM

如果把阶段推进、追问去重、决策关口、会话恢复都交给 LLM，短期看起来会更“聪明”，但很快会出现几个问题：

- 同一句输入在不同轮次、不同模型、不同温度下可能给出不同判断
- 自动化测试难以稳定回归，调一次 prompt 就可能连带打乱别的行为
- 多入口形态很难统一，CLI、网页、IDE skill 会逐渐各说各话
- 一旦需要审批、阻断、恢复、中断补偿，prompt 本身没有可靠执行力

所以这里不是“不相信 LLM”，而是：

`不把本该由产品运行时承担的职责，再伪装成语言模型的临场发挥。`

## 为什么也不能完全不用 LLM

反过来，如果把长尾语义、自然追问、模糊表达归因全部压给规则层，也会很快碰到上限：

- 角色关系会越来越难覆盖
- 用户口语表达会越来越像“没法归类”
- 继续卡和追问卡会越来越模板化
- 边界场景越补越多，最终陷入规则膨胀

所以本项目的方向不是“纯规则”和“纯 LLM”二选一，而是：

`让 LLM 处理高熵问题，让 runtime 处理低熵且必须稳定的问题。`

## 三类能力归属

### 第一类：必须由 Runtime 持有

这类能力要求稳定、可回归、可审计，不适合让 LLM 做最终裁决。

包括：

- 当前处于哪个阶段
- 当前 case 是否继续沿用
- 哪个决策关口仍然阻塞
- 哪些问题已经问过、答过、跳过
- 哪些问题属于同一问题家族
- 输出卡片属于哪一种骨架
- 中断、失败、恢复、继续这些终止语义
- runtime session、event log、execution ledger
- 审批、阻断、hook、工具调用闭环

判断标准：

- 这项能力一旦漂移，会不会直接破坏产品主线
- 这项能力是否需要用测试稳定锁住
- 这项能力是否需要留下工程真相，而不是模型意见

只要答案偏向“是”，就应该优先留在 runtime。

### 第二类：优先由 LLM 持有

这类能力主要依赖开放语义、自然表达和长尾理解，继续用规则硬堆收益会越来越差。

包括：

- 模糊口语输入的意图归因
- 一句话里隐含的角色关系理解
- 用户回答更像在补哪一个判断点
- 同一轮里多个补充点的轻度排序
- 更自然的追问改写
- 更贴近对话语气的继续卡表达
- 一句话可能对应的多个问题理解候选

判断标准：

- 这项能力是否高度依赖自然语言理解
- 这项能力是否存在大量合理表达变体
- 这项能力的目标是否更偏“像人协作”，而不是“结构正确”

只要答案偏向“是”，就应该优先让 LLM 承担主力。

### 第三类：适合混合判定

这是最重要的一类，因为 `PM Method Agent` 大概率会长期停在这里。

这类能力既需要语义弹性，又需要最终落到稳定字段和受控选项里。

包括：

- 回复解释器
- 角色关系抽取
- 决策倾向识别
- 追问焦点选择
- pre-framing 候选方向生成
- 继续分析时的阶段恢复

这类能力的推荐模式是：

- runtime 决定什么时候触发
- LLM 负责理解、改写、补全候选
- runtime 负责校验、裁剪、落盘和回退

## 当前项目里最重要的几条边界

### 1. 阶段推进不交给 LLM

例如：

- `context-alignment`
- `problem-definition`
- `decision-challenge`
- `validation-design`

这些阶段是产品主线，不是模型意见。

LLM 可以辅助判断“这轮补到的内容更像哪一阶段”，但最终的阶段恢复、关口切换和阻塞判定，必须由 runtime 负责。

### 2. 追问去重不交给 LLM 记忆

例如：

- “当前基线指标是什么”
- “当前的基线数据是多少”
- “目前这项行为的基线值大概在哪”

这些其实是同一个问题家族。

LLM 可以帮忙理解用户这句话是不是在回答这一类问题，但“既然已经回答过，就不要再追同家族问题”必须由 runtime 负责。

### 3. 决策关口不交给 LLM 直接放行

例如：

- 是否进入产品化阶段
- 是否优先评估非产品路径
- 是否暂缓

LLM 可以识别用户更偏向哪种表达，但最终仍然要收敛到受限选项集，并由 runtime 记录为正式关口状态。

### 4. 文案自然化可以逐步交给 LLM

例如：

- “这一轮现在先收什么”
- “为什么先补这几项”
- “补完后会往哪走”

这些能力的目标不是结构稳定，而是协作体验更自然。

它们很适合逐步交给 LLM，但前提是：

- 由 runtime 先提供焦点和原因
- LLM 只负责把它改写成更自然的卡片文案
- 如果 LLM 不可用，仍能回退到本地模板继续工作

## 一个简单判断法

后面每新增一项能力，都可以先问这三个问题：

### 1. 这项能力的错误，能不能接受成“偶尔说得不够好”

如果不能接受，比如会导致阶段乱跳、case 切错、审批绕过，那就不该主要交给 LLM。

### 2. 这项能力的目标，更像“判断正确”还是“表达自然”

如果更偏判断正确，优先 runtime。

如果更偏表达自然，优先 LLM。

### 3. 这项能力最后是否必须落到稳定字段、固定选项或可测试契约

如果必须落到这些受控结果里，就优先采用混合路径，而不是把最终真相直接交给 LLM。

## 当前推荐的能力归属表

| 能力 | 推荐归属 | 原因 |
| --- | --- | --- |
| 工作区 / case / project profile 状态 | Runtime 持有 | 属于状态真相，必须稳定落盘 |
| 阶段推进与阻塞关口 | Runtime 持有 | 属于方法主线，必须可回归 |
| 问题家族去重 | Runtime 持有 | 属于跨轮记账，不应依赖模型记忆 |
| 回复解释器 | 混合判定 | 语义理解交给 LLM，字段校验交给 runtime |
| 角色关系抽取 | 混合判定 | 高频可规则兜底，长尾由 LLM 补偿 |
| 决策倾向识别 | 混合判定 | 最终必须落到有限选项 |
| 追问焦点排序 | 混合判定 | 先由 runtime 控范围，再用 LLM 排序或改写 |
| 继续卡 / 追问卡文案 | LLM 优先 | 目标是更自然，不是更复杂的结构 |
| 模糊草稿的候选理解生成 | LLM 优先 | 高度依赖开放语义和候选发散 |

## 当前实现如何落位

按现在仓库的状态，已经比较明确的是：

- runtime 已经持有 case、workspace、project profile、runtime session
- runtime 已经持有阶段推进、关口控制、事件日志、审批和工具调用闭环
- runtime 已经开始持有问题家族去重和继续分析恢复点
- LLM 已经接入回复解释、前置收敛和部分文案增强的注入点
- 当前仍缺“LLM 参与追问自然化”和“LLM 参与追问焦点排序”的稳定产品化落点

也就是说：

`底座方向基本是对的，下一步不是把更多状态交给 LLM，而是把更需要人味的部分接进去。`

## 下一步最适合接入 LLM 的位置

如果后面按投入产出比排序，建议先做这三类增强：

### 1. 追问改写

输入：

- runtime 给出的追问焦点
- 当前阶段
- 用户上一轮原话
- 最近一条补充记忆

输出：

- 一句更自然的追问
- 一句为什么先追这个

约束：

- 不允许新增新的问题家族
- 不允许越过当前阶段边界
- 不允许改写成带答案暗示的引导性问题

### 2. 回答归因

输入：

- 用户这一轮原话
- 当前 pending questions
- 当前阶段

输出：

- 这句话更像回答了哪几个问题
- 置信度如何
- 是否存在“部分回答”

约束：

- 最终仍由 runtime 记录 answered questions
- LLM 结果必须可被本地规则或字段校验兜底

### 3. 继续卡文案增强

输入：

- runtime 已经确定的 follow-up focus
- 当前阻塞原因
- 最近补充的记忆摘要

输出：

- 更自然的“这轮先收什么”
- 更顺口的“为什么先补这项”
- 更轻的“补完后会往哪走”

约束：

- 只能改写表达，不能改写事实
- 不能新增新的阶段结论
- LLM 失败时直接回退到本地模板

## 暂时不建议接给 LLM 的位置

这几类能力目前先不要让 LLM 当主判：

- 新建 case 和继续 case 的最终裁决
- runtime policy 的阻断判断
- 审批状态变迁
- 工具调用是否允许执行
- event log / execution ledger 的正式写入
- 长会话压缩预算和恢复点维护

这些都是 agent runtime 的底座，不应该退化成“模型觉得差不多可以”。

## 反模式

后面继续做时，尽量避免这几种常见反模式：

### 1. 用 prompt 代替 runtime

例如把“不要重复追问”“别越过决策关口”“不要切错 case”全塞进 prompt。

这类规则应该由 runtime 明确执行，而不是让模型尽量记住。

### 2. 用规则代替 LLM 的长尾理解

例如为了覆盖几十种角色表达，把规则写成越来越长的字符串匹配表。

这类问题一旦进入长尾，应该考虑 LLM 补偿，而不是继续堆规则。

### 3. 让 LLM 同时负责理解、裁决和落盘

如果一项能力没有中间校验层，LLM 既做解释，又做选择，还直接改状态，后面很难收口，也很难查错。

## 和 Prompt 治理的关系

这份文档不替代 `prompt-layering.md`，两者分工不同：

- 这份文档回答“能力应该归谁”
- `prompt-layering.md` 回答“已经交给 LLM 的能力，prompt 该怎么分层治理”

也就是说，先决定：

- 这项能力应不应该交给 LLM

再决定：

- 交给 LLM 后，prompt 怎么组织、来源优先级怎么控制

## 和高级 Runtime 的关系

这份文档也不替代 `advanced-agent-runtime.md`。

两者关系是：

- `advanced-agent-runtime.md` 关注 query loop、事件流、恢复、预算、工具闭环
- 这份文档关注这些能力里，到底哪些部分由 runtime 主持，哪些部分可以让 LLM 参与

换句话说：

`这份文档定义的是分工边界，高级 runtime 文档定义的是运行机制。`

## 当前结论

沿着这个项目当前的方向，最合适的路线不是：

- 做一个“万能提示词”代理

也不是：

- 做一个完全不用 LLM 的规则系统

而是：

- 用 runtime 把产品主线、状态真相和工程约束守住
- 用 LLM 去提升语义理解、追问体验和长尾场景适配
- 用明确契约把二者连接起来

这会让 `PM Method Agent` 更像一个真正可持续演进的 agent 系统，而不是一段暂时能跑通的 prompt 工作流。
