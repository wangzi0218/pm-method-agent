# 规则分层与加载设计

## 目标

只靠一份 `agent.md` 或一段 prompt，不足以让系统真正适配不同团队和不同仓库。

一旦 agent 真的开始给团队干活，就会遇到这些局部制度：

- 公司有公司的协作规矩
- 仓库有仓库的开发约束
- 目录有目录的技术栈和交付方式
- 用户个人还有自己的表达偏好和工作习惯

这一层的目标，就是让系统能把这些局部制度吸收到运行时，而不是永远停留在演示环境里。

## 设计原则

### 1. 不只靠一种载体

更稳的方案不是三选一，而是：

- 用 `md` 表达语义规则
- 用 `json / yaml` 承接结构化策略
- 用 hook 执行确定性约束
- 用 runtime 做加载、合并和优先级控制

### 2. 规则表达和规则执行分开

规则文档负责回答：

- 这里默认怎么协作
- 哪些动作不鼓励
- 哪些输出更符合团队习惯

hook 负责回答：

- 哪些事必须被校验
- 哪些事违反了就要阻断
- 哪些动作要自动补执行

### 3. 规则必须分层

不同来源的规则不应混成一锅。

至少要区分：

- 产品默认规则
- 用户本地规则
- 项目级规则
- 目录级规则
- 本次任务追加规则

## 当前实现

当前仓库已经补上：

- `src/pm_method_agent/rule_loader.py`

并已经接入：

- `src/pm_method_agent/prompting.py`

这意味着：

- 系统已经能加载本地用户规则
- 系统已经能加载仓库根目录规则
- 系统已经能加载当前目录向上的 `.pmma` 规则
- 系统已经能把这些规则注入 prompt layer

## 当前支持的规则来源

### 1. 用户本地规则

默认位置：

- `~/.pmma/user-rules.md`

也支持显式指定：

- `PMMA_USER_RULES_PATH`

适合放：

- 个人输出偏好
- 个人协作习惯
- 不适合写进仓库的本地约束

### 2. 仓库级规则

当前支持这些文件名：

- `AGENTS.md`
- `PMMA.md`
- `agent.md`

适合放：

- 团队协作规范
- 仓库级说明
- 默认代码风格或工作方式

### 3. 目录级规则

当前支持：

- `.pmma/rules.md`
- `.pmma/project-rules.md`

系统会从当前目录一路向上查找，并逐层合并。

适合放：

- 某个子目录的技术栈偏好
- 该目录特有的交付要求
- 局部目录的测试、命名或结构约束

### 4. 结构化策略文件

当前支持：

- `.pmma/policy.json`

适合放：

- `behavior_rules`
- `tool_constraints`
- `output_discipline`
- `project_instructions`
- `custom_append`

之所以第一版先用 `json`，是因为：

- 无额外依赖
- 当前足够表达最小结构化策略
- 后续再扩展成 `yaml` 也比较顺

### 5. 环境变量追加

当前支持：

- `PMMA_PROMPT_PROJECT`
- `PMMA_PROMPT_APPEND`

这层更适合做：

- 当前环境追加
- 临时项目注入
- 不想落盘的试验性规则

## 当前优先级

第一版当前的加载顺序是：

1. 用户本地规则
2. 仓库级规则
3. 目录级规则
4. 结构化策略文件
5. 代码里当前任务追加的规则
6. 环境变量追加

这里的意思不是“谁更重要”，而是当前实现里的合并顺序。

从长期设计上，更合理的优先级仍然是：

1. runtime 硬约束
2. 本次任务追加规则
3. 目录规则
4. 项目规则
5. 用户偏好
6. 产品默认规则

第一版还没有把所有优先级都工程化到 runtime policy 里，但至少已经把规则来源分开了。

## 当前接入边界

这一版规则加载器当前只接到了 prompt layer。

它还没有直接控制：

- runtime 生命周期
- 危险动作拦截
- hook 执行
- 测试强制策略
- 目录级命令白名单

也就是说：

当前这一步先解决的是“规则能被看见和吸收”，
还没有完全解决“规则一定被执行”。

不过当前仓库已经开始补最小 `runtime policy` 层，用来承接其中一部分硬约束。

## 如何检查当前规则

现在已经可以直接通过 CLI 查看当前目录实际生效的规则层和策略层：

```bash
PYTHONPATH=src python3 -m pm_method_agent.cli rules \
  --base-dir . \
  --show-prompt
```

这个入口适合用来确认几件事：

- 当前到底加载了哪些规则文件
- 当前 prompt 最终吃进了哪些行为规则、工具约束和输出纪律
- 当前 runtime policy 的有效值是什么

这样后面无论是接 `AGENTS.md`、`.pmma/rules.md`，还是接更细的目录策略，都先能看见，再调整。

## hook 应该放在哪

hook 不适合用来表达规则本身，更适合做这些事：

- 危险动作校验
- 提交前检查
- 指定目录的自动测试
- 配置变更的 schema 校验
- 审计记录

更合理的长期结构是：

1. 规则加载器负责吸收规则
2. prompt layer 负责让模型理解这些规则
3. runtime policy 负责让系统判断哪些规则属于硬约束
4. hook 负责执行其中可判定、可阻断的部分

## 下一步

这层后续更适合继续补：

1. `yaml` 规则文件支持
2. 目录作用域与规则来源的更明确优先级
3. 更完整的 runtime policy 层
4. hook enforcement 层
5. 针对 `sub-agent` 的目录和角色规则传播
