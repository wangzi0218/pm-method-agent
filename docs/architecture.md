# 架构设计

## 目标

`PM Method Agent` 的架构设计需要同时满足三件事：

- 第一版足够简单、稳定、便于快速迭代
- 用户看到的是一个统一的方法产品，而不是很多碎片化能力
- 内部能力后续可以升级成复合智能体系统，但不打破已有接口

## 推荐的 V1 形态

### 对外形态

第一版只暴露一个用户入口：

- `pm-method-agent`

用户不应该自己在十几个方法能力之间做选择，这些复杂度应由系统内部吸收。

从长期看，这个项目应当允许同一套方法内核被多种入口复用：

- 命令行工具
- 主代理 / AI 工具入口
- 可选网页入口
- 本地 HTTP 服务
- 后续 MCP 适配层

用户可以从不同界面进入，但不应该面对不同的方法逻辑。

### 对内形态

V1 采用“单一编排器 + 结构化分析模块”。

核心组件：

1. 编排器
负责会话流转、阶段路由、案例状态维护、分析模块调用、结果合并和决策关口触发。

2. 场景信息输入层
负责收集和补齐最小场景信息，例如企业产品/消费者产品、桌面端/移动端、渠道、部署形态、核心用户角色等。

3. 分析模块
职责边界清晰的方法模块。它们不直接拥有对话，只读取当前案例状态，并输出结构化结论项。

4. 决策关口引擎
决定系统能否自动继续，还是必须由人类做阶段级判断。

5. 输出渲染器
把结构化结论项渲染成用户可读的审查卡、摘要和下一步建议。

6. 交付适配层
把同一套方法内核挂接到单一入口能力、命令行工具或网页界面，而不改变底层逻辑。

7. 服务层
负责会话读写、状态承接、阶段继续执行，以及未来网页和 API 所需的调用封装。

## 为什么一开始不做复合智能体

第一版直接做多自主智能体，成本通常高于收益：

- 重复推理会增多
- 输出冲突会增多
- 语气和结构更难稳定
- 回归评测更难做
- 提示词和状态管理复杂度更高

对于一个方法型产品来说，第一阶段“稳定一致”比“高度自治”更重要。

## 为什么必须引入场景信息输入层

你提到的这个点是关键：不同产品场景下，能成立的方法判断并不一样。

例如：

- 企业产品和消费者产品的价值判断逻辑不同
- 桌面端与移动端、小程序、嵌入式终端的展示空间和交互成本不同
- 公域分发和私有部署产品，渠道限制不同
- 面向老板、运营、一线执行者和普通消费者时，关键角色结构完全不同

如果不先对齐这些基础信息，很多“看起来合理”的判断其实可能是南辕北辙的。

所以，方法层不应该假装自己天然知道上下文，而应当：

- 在分析前先建立一个轻量的场景基础信息
- 只补充真正影响判断的场景信息
- 避免让用户填一大堆低价值表单

## 场景基础信息的设计原则

场景基础信息不应是一个沉重的需求模板，而应是一个最小、可渐进补充的场景定义层。

建议优先收集的字段：

- `business_model`：企业产品、消费者产品、内部产品
- `primary_platform`：桌面端、移动网页、原生应用、小程序、多端
- `distribution_channel`：如私有部署、软件服务化交付、公域应用市场、微信生态等
- `product_domain`：如医疗、教育、电商、内容、企业协同
- `target_user_roles`：谁提需求、谁使用、谁承担结果
- `constraints`：合规、设备能力、交付周期、组织资源等

原则上，系统只有在缺少这些信息会明显影响判断时，才应该继续追问。

## 升级路径

系统从一开始就要保证：内部分析模块未来可以被提升为专项智能体。

### 必须保持稳定的接口

- 案例状态
- 场景基础信息
- 分析模块输出契约
- 决策关口契约
- 面向用户的阶段模型

### 形态演进

#### 第一阶段：顺序执行的轻量运行时

- 编排器顺序调用分析模块
- 分析模块都是内部模块
- 对外只有一个统一入口

#### 第二阶段：可插拔分析模块运行时

- 分析模块变成可注册、可替换的单元
- 编排器可以按阶段跳过或并行执行分析模块
- 分析模块的提示词或具体实现可以变化，但输出契约不变

#### 第三阶段：复合智能体运行时

- 部分分析模块升级成半自主专项智能体
- 编排器把任务委派给它们
- 专项智能体仍返回同一套结构化结论项

也就是说，升级是架构层升级，而不是产品概念重做。用户入口可以保持不变。

## 交付适配策略

为了保留未来灵活性，方法核心必须与交付外壳分离。

推荐分层：

1. 方法核心运行时
负责阶段路由、契约、分析模块协调、决策逻辑和场景信息处理。

2. 服务层
负责会话状态持久化、案例读取与更新、阶段推进、重跑控制，以及未来 HTTP API 所需的统一调用接口。

3. 交付适配层
负责把用户输入转成案例状态更新，把决策关口转成各类界面中的交互动作。

4. 展示外壳
最终用户看到的外壳，例如单一入口能力、命令行工具或网页工作台。

这种分层允许你：

- 先用命令行工具做评测和开发调试
- 再挂接一个主代理入口
- 后续再挂到网页界面上做结构化表单和决策卡片
- 最后再开放 API，而不重写方法内核

变化的应该是外壳，不应该是方法内核。

## 为什么当前先做 HTTP，而不是直接押注单一 AI 协议

HTTP 在这个项目里首先承担的是“统一服务层”角色，而不是最终用户入口。

当前先做 HTTP，主要是因为：

- CLI、skill、agent、网页都能直接复用
- 本地调试和回归验证成本最低
- 不会把方法内核绑死在某一种 AI 宿主协议上

更合适的长期结构应当是：

- 方法核心
- 会话服务层
- HTTP 服务
- MCP 或其他 AI 原生接入层
- skill / agent / 网页外壳

也就是说，HTTP 不是和 AI 原生方案冲突，而是它们的稳定底座。

## 为什么网页形态仍然需要 API

如果未来提供交互网页，本质上仍然需要一层服务接口。

原因是网页本身不适合直接承担这些职责：

- 保存案例状态
- 维护多轮会话
- 控制阶段推进
- 调用模型或内部分析能力
- 处理鉴权、限流和日志

因此，即使最初不对第三方开放平台，也应尽早把内部服务接口抽象出来。

建议理解为：

- 先有方法核心
- 再有服务层
- CLI、网页、主代理都调用同一套服务层
- HTTP API 只是服务层对外暴露的一种外壳

## 面向 CLI、网页与 API 的统一结构

建议未来稳定为三层：

### 1. `core`

负责：

- `case state`
- 阶段状态机
- 分析模块
- 决策关口
- 输出契约

### 2. `service`

负责：

- 创建案例
- 读取与更新案例
- 合并用户补充信息
- 从阻塞阶段继续执行
- 提供统一的调用接口给 CLI、网页和 agent

### 3. `interfaces`

负责不同交付形态：

- `cli`
- `web`
- `agent`
- `http api`

这意味着未来无论用户在命令行还是网页里使用，本质上都在调用同一套方法系统。

## 未来网页 / API 的最低能力

如果后续进入网页交付阶段，建议至少支持这些接口：

- `POST /cases`
  创建新案例并返回首张卡片
- `GET /cases/{id}`
  获取当前案例状态与最新卡片
- `POST /cases/{id}/reply`
  用户补充回答后继续推进
- `POST /cases/{id}/rerun`
  基于更新后的上下文重跑某阶段
- `GET /cases/{id}/history`
  查看阶段推进历史

这些接口不要求现在就实现，但当前架构应该为此预留稳定口子。

## 默认输出形态

在当前阶段，最适合这个产品的默认输出形态是“轻量结构化审查卡”。

原因是：

- 用户大概率在 IDE、命令行工具或 AI 对话窗口中调用
- 他们需要快速看懂并继续推进，而不是阅读一份完整报告
- 太随意会退化成聊天，太正式又会抬高使用门槛

因此，建议把默认输出稳定在“专业协作卡片”这一层，而不是“正式报告”。

## 建议的分析模块分组

为了减少碎片化，原来的十个原子能力建议收敛为四个内部模块。

### 1. `problem-framing`

包含：

- 现象 / 解释 / 方案分离
- 关键角色映射
- 替代方案分析
- 初步问题强度判断

### 2. `root-cause-and-alternatives`

包含：

- 类“五问法”的根因探查
- 非产品解法挑战
- why-now 时机判断

### 3. `decision-challenge`

包含：

- 反方论证
- 范围与非目标
- 是否值得做的挑战

### 4. `validation-design`

包含：

- 可证伪假设编写
- 最小验证动作设计
- 成功指标与护栏指标设计

## 面向用户的阶段模型

用户体验应当按阶段组织，而不是按碎片能力组织。

### 阶段 0：场景对齐

目标：
先确认当前产品与场景的最小背景信息是否足够支撑判断。

### 阶段一：问题定义

目标：
判断当前描述是否已经是一个足够明确的问题，而不只是方案表达。

### 阶段二：决策挑战

目标：
判断这个问题是否值得现在解决，以及是否应该优先产品化。

### 阶段三：验证设计

目标：
定义最小验证路径，并提前约定成功和停止标准。

## 人工决策策略

系统不应该在每个不确定项上都向人确认。

人工输入应当只保留给那些会影响阶段推进的判断，例如：

- 当前场景信息是否已经足够
- 当前证据是否已足够进入方案设计
- 是否应该优先尝试非产品路径
- 当前问题是继续推进，还是应该延后或放弃

## V1 的非目标

- 全自动外部调研和联网研究
- 一个庞大的复合智能体生态
- 一个一开始就很重的网页平台
- 深度垂直业务知识库

第一版的目标应该是先证明：方法层本身确实能创造区别于通用 AI 对话工具的价值。
