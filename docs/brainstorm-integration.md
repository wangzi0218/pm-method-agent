# Brainstorm 融合设计

## 文档目标

这份文档回答一个具体问题：

`PM Method Agent` 是否需要吸收 `brainstorm` 能力，如果需要，应当以什么方式吸收。

这里讨论的不是“要不要让系统更会聊天”，而是：

- 在用户输入还很模糊时，系统如何先做有限发散
- 发散结果如何服务后续的问题定义与决策判断
- 如何避免产品退化成普通聊天助手

如果要继续往实现收敛，可继续看：

- [docs/brainstorm-minimal-design.md](/Users/wannz/Documents/sourcetree/pm-method-agent/docs/brainstorm-minimal-design.md)

## 结论先行

可以吸收，而且值得吸收。

但更合适的做法不是增加一个独立平行的 `brainstorm agent`，而是：

`把 brainstorm 作为主代理内部的前置思考层。`

它的职责应当是：

- 帮用户把模糊输入拆成几种可能的问题理解
- 暂时拉开视角，避免过早顺着单一方案往下走
- 帮主代理决定下一步更值得问什么

它不应替代：

- 问题定义
- 决策挑战
- 验证设计

## 为什么现在要补这一层

当前主代理已经具备：

- 状态推进
- 多轮承接
- 项目背景记忆
- 决策关口

但从真实试跑看，仍然存在两个体验问题：

### 1. 太快进入收敛

用户刚补完背景，系统就容易直接进入完整审查卡。

这会让交互显得像一个“流程化分析器”，而不是一个会协作推进的代理。

### 2. 输入还没站稳时，系统缺少一个“先拉开看看”的动作

很多真实输入其实同时混着：

- 现象
- 解释
- 方案倾向
- 角色冲突
- 紧迫性感受

如果没有一层轻量发散，系统就容易：

- 过早把一句话认成一个明确问题
- 过早给出收敛性的判断
- 错过其他同样合理的理解路径

## 不应该怎么做

在设计这一层时，有三种常见但不合适的方向。

### 1. 不要把它做成独立主流程

如果把 `brainstorm` 做成一条完整平行流程，用户会额外理解：

- 什么时候该用 brainstorm
- 什么时候该用审查
- 两者之间如何切换

这会明显增加使用负担。

### 2. 不要无限发散

如果让系统围绕一个模糊输入持续发散，很容易退化成：

- 多轮陪聊
- 列一堆角度
- 但不推进真实判断

这和 `PM Method Agent` 的定位冲突。

### 3. 不要把发散结果直接当结论

`brainstorm` 输出的只是候选理解，不是最终判断。

如果直接把候选路径写进结论层，后续卡片会显得前提不稳、口径混乱。

## 这一层应该扮演什么角色

更准确地说，`brainstorm` 应该是：

- 主代理内部的 `pre-framing layer`
- 一种在高不确定性输入下自动触发的“有限发散模式”

它的目标只有一个：

`在正式进入问题定义前，先帮用户把问题空间收拢到几个更靠谱的方向。`

## 适合触发的场景

建议只在以下几类输入中触发。

### 1. 用户输入明显模糊

例如：

- “最近前台总说这里不太顺。”
- “我感觉这个地方是不是该做点什么。”
- “这个需求大家都在提，但我没想清楚值不值得。”

### 2. 输入同时混着问题和方案

例如：

- “想加一个提醒入口，感觉现在容易漏。”
- “是不是做个新手引导浮层，提升发帖率。”

### 3. 同一句话里存在多种合理理解

例如：

- 是流程问题，还是产品能力问题
- 是角色协作问题，还是单点操作问题
- 是紧急业务问题，还是低优先级体验问题

### 4. 用户显式要求一起想想

例如：

- “你帮我一起发散一下。”
- “还有哪些可能没想到？”
- “这件事可能有哪些理解角度？”

## 不适合触发的场景

以下场景应直接进入原有流程，不需要额外 `brainstorm`。

### 1. 场景信息已经足够清楚

如果：

- 产品类型明确
- 平台明确
- 角色分工明确
- 当前问题也已经接近事实描述

这时再发散只会拖慢节奏。

### 2. 用户明确要求做收敛判断

例如：

- “直接帮我判断值不值得做”
- “直接看应不应该产品化”

这类输入可以允许系统跳过 `brainstorm`。

### 3. 已经处在后续阶段

如果当前 case 已进入：

- 决策挑战
- 验证设计
- 暂缓/阻塞后的恢复

原则上不再自动回到 `brainstorm`，除非用户明确要求重开理解路径。

## 这一层的输出边界

`brainstorm` 输出应严格控制在“有限发散”范围内。

建议固定为四个槽位：

### 1. 当前可能在说的几个问题方向

数量建议：

- 2 到 4 个

例如：

- 可能是前台提醒流程不稳定
- 可能是复诊患者识别不稳定
- 可能是责任分工不清导致的漏提醒

### 2. 每个方向背后的关键假设

例如：

- 如果是流程问题，假设当前主要靠人工记忆
- 如果是责任分工问题，假设前台和店长对“谁该提醒”认知不一致

### 3. 最值得先确认的信息缺口

数量建议：

- 2 到 3 个

例如：

- 现在到底是谁在操作提醒
- 漏提醒具体发生在哪个环节
- 这件事最近为什么被放大了

### 4. 建议先沿哪条继续

这一项很关键。

系统不能只发散不收口，必须告诉用户：

- 现在最值得先补哪一类信息
- 更建议从哪个问题方向继续

## 建议的用户可见输出形态

不建议直接叫“brainstorm 卡”，否则太像一个额外产品。

更自然的呈现方式可以是：

- `继续卡`
- `先一起收一收`
- `先别急着定，当前更像这几种情况`

其中一版更贴近当前产品语气的结构可以是：

### 当前理解

- 这句话现在至少有 2 到 3 种可能的理解

### 更像哪几类问题

- 方向 1
- 方向 2
- 方向 3

### 先确认这几件事

- 问题 1
- 问题 2
- 问题 3

### 更建议先往哪条继续

- 建议先补 `流程现状`
- 或建议先补 `角色分工`

## 中间产物契约

`brainstorm` 不应只生成文本。

如果后续真的落代码，建议补一层结构化中间产物，例如：

```json
{
  "brainstorm_candidates": [
    {
      "candidate_id": "B-001",
      "label": "提醒流程不稳定",
      "summary": "当前更像人工流程执行不稳定，而不是单点功能缺失。",
      "assumptions": [
        "当前提醒主要靠人工触发",
        "提醒责任没有稳定落到某个角色"
      ],
      "confidence": "medium"
    }
  ],
  "recommended_candidate_id": "B-001",
  "priority_questions": [
    "谁在日常流程里实际操作提醒",
    "漏提醒发生在哪个环节",
    "最近为什么开始变得更重要"
  ]
}
```

这层中间产物应满足两点：

- 后续分析器可以读，但不会把它当结论
- 用户如果不同意某个候选方向，可以直接修正

## 和当前状态机的关系

更合适的接法不是新增一条完整大流程，而是在 `intake` 和 `problem_definition` 之间增加一个可选层：

```text
intake
  -> brainstorm? 
  -> context_alignment
  -> problem_definition
  -> decision_challenge
  -> validation_design
```

### 建议规则

- `intake` 后先判断是否需要 `brainstorm`
- 如果需要，就先输出一张轻量继续卡
- 用户补完关键方向后，再进入 `problem_definition`

这样可以保留当前主流程稳定性，又不会让系统太快进入“出报告模式”。

## 和 skill / agent 的关系

如果未来这个项目既能作为 `skill` 使用，也能作为 `agent` 使用，那么 `brainstorm` 更适合作为：

- 主代理内部的一个动作
- 而不是用户显式选择的独立模式

更自然的用户体验应该是：

用户只是在说一句模糊话，
系统判断这时候先发散一下更合适，
然后给出一张有限发散、可继续推进的卡。

而不是让用户先学会：

- “我要调用 brainstorm”
- “我要先 brainstorm 再审查”

## 对当前实现的直接启发

即使暂时不写代码，也已经可以先吸收三条原则。

### 1. 补完背景后不要立刻给完整审查卡

应该优先给一张更轻的继续卡，引导用户先补关键缺口。

### 2. 追问前先判断“是否存在多个合理理解”

如果有，就不要急着直接下一个硬判断。

### 3. 输出要始终带一个收敛建议

不能只列可能性，必须告诉用户：

- 先补哪件事
- 先沿哪条问题理解继续

## 实现建议

如果后面决定正式落代码，建议按这个顺序推进：

### 第一步：文档收敛

明确：

- 触发条件
- 输出契约
- 与现有状态机的关系

### 第二步：先做轻量版本

先不要接 LLM，多用启发式规则验证：

- 哪些输入该先发散
- 发散 2 到 3 个方向是否已经够用

### 第三步：再决定是否接 LLM

只有当启发式版本明显不够时，再让 LLM 参与：

- 候选方向生成
- 候选方向排序
- 更自然的继续卡文案

### 第四步：把它纳入评测集

新增评测维度：

- 是否过早收敛
- 是否发散过度
- 是否给出明确收敛建议

## 一句话总结

`brainstorm` 值得融入，但它在这个项目里最合适的位置，不是一个独立入口，而是一层“有限发散、立即收口”的前置思考能力。`
