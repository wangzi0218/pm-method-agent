# 交互层与记忆层设计

## 目标

当前项目已经具备：

- 方法内核
- 多轮会话状态机
- LLM 适配层
- 本地 HTTP 服务

下一阶段真正影响可用性的，不再只是分析模块本身，而是：

- 用户如何触发
- 用户如何持续使用
- 系统如何给建议而不是只给结果
- 多轮互动如何自然承接
- 用户偏好、项目背景和长期上下文如何沉淀

这一层决定 `PM Method Agent` 最终更像“长期协作 agent”，还是“偶尔跑一次的分析工具”。

## 一句话定义

下一阶段的核心，不是继续增加更多分析模块，而是：

`把方法运行时升级成可长期协作的交互系统。`

## 设计原则

### 1. 入口要少，触发要自然

用户不应先理解内部模块和状态机，再决定怎么使用。

更合适的体验是：

- 用户表达一个意图
- 系统判断当前应该创建新 case、继续旧 case，还是只补充背景

这里的“表达一个意图”不应被理解为只能说标准命令句。

系统应当能接住：

- 一个点子
- 一句抱怨
- 一段很粗的草稿
- 一个带着方案倾向的半成品需求
- 一个还没想清楚的方向描述

### 2. 输出不只回答问题，还要推进下一步

系统不应只是“分析完就结束”，而应明确给出：

- 现在建议做什么
- 还差什么
- 是否值得继续
- 下一轮最值得补哪一类信息

### 3. 记忆应渐进沉淀，而不是一开始强迫填写

用户不应该先维护一堆 profile 才能开始使用。

更合适的方式是：

- 先开始使用
- 在真实互动中逐步沉淀偏好和背景
- 只在显著提升判断质量时，才显式提示用户补充或确认

### 4. 记忆必须分层

不能把所有上下文都混成一个大文本。

至少要区分：

- 用户偏好
- 项目背景
- 当前案例上下文

### 5. 主代理要显式知道自己在做什么

主代理不只是“接一句回一句”，而要清楚当前动作属于哪类交互：

- 新建分析
- 继续追问
- 响应补充
- 推荐下一步
- 读取背景
- 更新长期记忆

## 用户触发设计

### 典型触发意图

建议先收敛为以下几类：

但触发识别不应只依赖显式命令句，而应优先判断用户当前更像是在：

- 提出一个新问题
- 延续上一个问题
- 补背景
- 求建议
- 查历史

### 1. 新建分析

例如：

- “帮我审查这个需求”
- “帮我判断这个需求值不值得做”
- “帮我看一下这个场景现在该不该推进”
- “我有个想法，但感觉还不太成型”
- “最近总有人提这个需求，我也没想清楚值不值得做”
- “先记个草稿：前台想加一个提醒入口”

系统动作：

- 创建新 case
- 判断是否补场景信息
- 输出第一张卡片

### 2. 继续分析

例如：

- “继续刚才那个 case”
- “基于上面的信息继续”
- “补充一下这个需求的现状”
- “还有个背景你要一起算进去”
- “刚才那个我再补一句”
- “我想了一下，这里不是运营问题”

系统动作：

- 读取当前 case
- 承接上一阶段继续推进

### 3. 补项目背景

例如：

- “这个项目是 ToB 医疗服务平台”
- “我们主要是移动端场景”
- “这个团队上线周期很紧”
- “这个产品一直跑在诊所前台场景”
- “我们团队默认先看到诊率和履约率”

系统动作：

- 不一定立即推进 case
- 优先写入项目记忆或案例背景

### 4. 读取建议

例如：

- “我现在下一步该做什么”
- “还有哪些信息最值得补”
- “这个 case 现在卡在哪”
- “如果我只补一件事，最该补什么”
- “现在先继续还是先停一下”

系统动作：

- 优先返回建议和阻塞点
- 不重新做整轮分析

### 5. 读取历史

例如：

- “看看之前的判断”
- “这个 case 做过哪些决定”
- “我之前补过哪些信息”

## 触发识别原则

### 1. 优先识别任务，不优先识别格式

用户说得不标准，不代表没有明确意图。

系统应先判断这句话更像哪一类任务，而不是先要求用户改写成命令。

### 2. 允许从草稿进入分析

如果用户只给了一个模糊点子，系统也应能：

- 先创建 case
- 标记当前输入强度较弱
- 输出更偏问题定义和场景补充的第一张卡片

### 3. 允许从抱怨和现象进入分析

例如：

- “前台最近老是漏提醒患者”
- “新用户发帖还是起不来”
- “管理者总说这个流程太绕”

这类输入即使没有明确说“帮我审查”，也应默认进入新建分析，而不是要求用户先整理成需求文档。

### 4. 用户显式要求“先记录”

如果用户更像是在留草稿，而不是要求立即分析，系统可先：

- 创建一个轻量 case 或草稿状态
- 记录当前想法
- 告诉用户后续可继续补充并进入正式审查

系统动作：

- 返回当前案例历史和阶段轨迹

## 持续互动设计

### 交互单元

建议把用户与系统的一次有效互动分成四种：

### 1. 输入型互动

用户提供原始需求、补充证据、补充背景。

### 2. 决策型互动

用户对关口做选择。

### 3. 校准型互动

用户修正系统判断，例如：

- “不是这个角色在用”
- “这个问题已经试过流程解法”
- “这里不是增长目标，是合规目标”

### 4. 建议型互动

用户不补信息，而是想知道现在最合适的动作。

## 系统主动建议设计

系统的建议不应只有“去补更多信息”这一种。

建议至少分为四类：

### 1. 推进建议

例如：

- 现在可进入验证设计
- 现在不建议继续产品化
- 当前应先做非产品路径试行

### 2. 补充建议

例如：

- 先补现状流程
- 先补角色冲突
- 先补当前基线指标

### 3. 纠偏建议

例如：

- 当前输入混入了方案表达
- 当前问题仍然定义过宽
- 当前把时机判断和方案判断混在一起了

### 4. 记忆建议

例如：

- 这类项目信息适合记录为项目背景
- 这项偏好建议保存为用户偏好
- 这条约束建议写入当前工作空间记忆

## 记忆层设计

### 三层记忆模型

建议先定义三层：

### 1. `user profile`

表示这个用户长期偏好的交互与判断习惯。

建议字段：

- `preferred_output_style`
- `preferred_language`
- `decision_style`
- `frequent_product_domains`
- `common_constraints`

适合记录的内容：

- 偏好卡片简洁还是更完整
- 偏好中文还是双语
- 是否更常做 ToB / ToC / 增长 / 流程型问题

### 2. `project profile`

表示某个项目或产品线的长期背景。

建议字段：

- `project_name`
- `business_model`
- `primary_platform`
- `product_domain`
- `distribution_channel`
- `core_user_roles`
- `stable_constraints`
- `success_metrics`

适合记录的内容：

- 产品类型和平台
- 长期角色结构
- 合规、设备、交付周期等稳定限制
- 团队常看的核心指标

### 3. `workspace memory`

表示某一段时间内的工作上下文，不一定和单个 case 完全绑定。

建议字段：

- `active_cases`
- `recent_decisions`
- `recent_assumptions`
- `open_questions`
- `linked_project_profile`

适合记录的内容：

- 当前在推进哪些案例
- 最近做过哪些阶段级判断
- 哪些假设仍未验证

## 记忆写入原则

不是所有用户说过的话都应该自动写入记忆。

建议按以下规则处理：

### 自动写入

适合自动沉淀的内容：

- 重复出现的项目背景
- 明确表达的稳定约束
- 已经多次使用的输出偏好

### 候选写入

系统可以提示用户确认后再写入：

- 可能长期成立的项目规则
- 用户表达出来的风格偏好
- 团队常看的指标口径

当前版本建议先把这层做轻：

- 不急着加单独的确认面板
- 先在卡片或工作区里给一句提示
- 只告诉用户“这句更像项目背景 / 使用偏好 / 当前案例”

例如：

- “我们团队默认先看诊率和履约率” 更像项目背景
- “我更喜欢先给结论再展开” 更像使用偏好
- “这次前台昨天漏了 6 次提醒” 更像当前案例

### 不写入长期记忆

不适合写入长期记忆的内容：

- 单个 case 的偶然现象
- 未确认的猜测
- 只在当前轮次有效的临时表达

## 和当前运行时的关系

这一层不应替代当前已有的 `case state`。

更准确的关系应是：

- `case state`：负责单个案例当前分析状态
- `workspace memory`：负责一段工作会话里的共享上下文
- `project profile`：负责项目长期背景
- `user profile`：负责用户长期偏好

## 推荐的落地顺序

为了避免一次做太重，建议按顺序推进：

1. 先定义触发类型和建议类型。
2. 再补最小 `project profile`。
3. 再补 `user profile` 的少量字段。
4. 最后再做更丰富的 `workspace memory`。

## 当前已落地

当前仓库已经开始落地其中一部分：

- `project profile` 的最小数据结构与本地存储
- `workspace` 的最小状态
- 统一入口的最小 `agent shell runtime`
- `user profile` 的最小字段与工作区级存储入口
- 工作区概览里的三层记忆展示：
  - 当前项目背景
  - 当前使用偏好
  - 当前工作记忆

这意味着当前已经能：

- 记住当前活跃案例
- 记住当前工作区绑定的项目背景
- 在一句自然语言补充里默认承接当前上下文
- 通过工作区命令或 HTTP 接口写入最小用户偏好
- 在工作区概览里看见系统现在记住了什么

## 对 skill / agent 的意义

如果未来主要形态是 skill / agent，那么这份设计决定了三个关键问题：

1. 用户怎么开场最自然
2. 主代理如何判断“这是新 case 还是继续旧 case”
3. 系统如何在多轮中显得像一个长期协作者，而不是每次都重头问起
