# LLM 边界场景设计

## 文档目标

这份文档回答的不是“要不要接 LLM”，而是：

`在 PM Method Agent 里，哪些能力必须留在本地工程层，哪些能力应该优先交给 LLM，哪些能力适合走混合路径。`

如果这一层边界不先收紧，项目很容易滑向两个极端：

- 全靠 LLM 临场发挥，产品主线越来越不稳定
- 明明应该交给 LLM 的事，继续用越来越重的规则硬撑

这两种方向都不适合这个项目。

## 先说结论

更合理的分工不是：

- 本地规则和 LLM 二选一

而是：

- 本地工程层负责稳定骨架
- LLM 负责高弹性理解与表达增强
- 两者之间通过清晰契约协作

一句话概括：

`本地层保证系统像一个产品，LLM 让它更像一个会协作的智能体。`

## 分层原则

建议始终按三个标准判断一项能力该放哪一层：

### 1. 这项能力是否要求稳定可回归

如果答案是“是”，优先留在本地层。

典型例子：

- 状态机推进
- 输出卡片结构
- 阻塞与放行规则
- case / workspace 记忆

### 2. 这项能力是否高度依赖开放语义理解

如果答案是“是”，优先交给 LLM。

典型例子：

- 复杂自然语言意图识别
- 模糊表达的候选理解生成
- 更自然的追问与继续卡文案

### 3. 这项能力是否既要稳定，又需要语义弹性

这种情况更适合走混合路径。

典型例子：

- 回复解释
- 角色关系抽取
- pre-framing 候选方向生成

## 必须留在本地工程层的能力

这些能力不建议直接交给 LLM 主导。

### 1. 状态机和阶段推进

包括：

- 当前处于哪个阶段
- 是否进入 `pre_framing`
- 是否进入 `context_alignment`
- 是否停在 `decision_challenge`
- 是否进入 `validation_design`

原因：

- 这是产品主线，不是模型意见
- 必须稳定可测
- 一旦漂移，整个产品体验会散掉

### 2. 输出契约和卡片骨架

包括：

- `continue-guidance-card`
- `context-question-card`
- `decision-gate-card`
- `review-card`

原因：

- 用户看到的是产品结构，不是模型自由发挥
- GitHub 项目也需要稳定输出契约
- 自动化测试依赖这层稳定存在

### 3. case / workspace 持久化与上下文合并

包括：

- 会话历史
- 当前活跃 case
- project profile
- note bucket
- stage history

原因：

- 这是工程责任，不是推理责任
- 不应让模型决定状态真相

### 4. 决策关口的触发机制

包括：

- 什么时候必须让人类做决定
- 哪些关口是阻塞的
- 哪些关口只是提示

原因：

- 这是方法约束
- 不应该被模型一句“看起来可以继续”随意覆盖

### 5. 回退、失败处理和保底行为

包括：

- LLM 超时或失败时如何回退
- 非法输出如何兜底
- 解释器切换逻辑

原因：

- 这是系统可靠性边界
- 不应再交给 LLM 处理

## 应优先交给 LLM 的能力

这些能力如果长期继续用规则硬堆，收益会越来越低。

### 1. 高弹性自然语言理解

包括：

- 用户意图识别
- 模糊口语表达理解
- 更复杂的上下文提取
- 不同表达风格的归一化理解

例如：

- “我感觉这块是不是该看看”
- “这事运营那边最近一直在催”
- “老板开始盯这个结果了”

这类表达很难靠规则长期覆盖。

### 2. pre-framing 候选方向生成

包括：

- 一句话可能对应的 2 到 3 条问题理解
- 每条理解背后的关键假设
- 更像哪条的排序

原因：

- 这类任务天然依赖语义弹性
- 规则可以做最低保底，但不适合长期承担主力

### 3. 更自然的协作文案

包括：

- 继续卡文案
- 追问表达
- 关口提示语气

原因：

- 这类能力的目标不是“能输出”，而是“读起来像自然协作”
- 如果长期靠模板拼，会很快显得机械

### 4. 复杂角色关系抽取

尤其是这些情况：

- 提出者、使用者、结果责任人不是同一拨人
- 一句话里同时出现组织角色和用户角色
- 存在隐含责任关系

例如：

- “运营提的，一线审核同学在处理，老板只盯结果。”
- “医生不直接操作，但会对门店执行效果有意见。”

这类场景很适合让 LLM 补强。

## 适合走混合路径的能力

这是最关键的一层，因为项目后面大概率会长期停在这里。

### 1. 回复解释器

建议结构：

- 本地层负责结构化字段和兜底
- LLM 负责理解回复内容
- 最终结果必须走本地校验

这意味着：

- LLM 可以生成 `context_updates`
- 但本地层要校验字段是否合法

### 2. 角色抽取

建议结构：

- 本地规则覆盖高频角色和高频句式
- LLM 负责复杂关系补充
- 最终结果仍以标准化角色字段写回 case

这样做的好处是：

- 高频场景稳定
- 长尾场景不至于完全失效

### 3. pre-framing

建议结构：

- 本地层负责判断是否触发
- LLM 负责生成和排序候选方向
- 本地层负责控制数量、阶段和卡片骨架

这是一个很典型的“LLM 做脑力，本地层做调度”的场景。

### 4. 决策倾向识别

比如识别：

- 暂缓
- 优先非产品路径
- 继续产品化

建议结构：

- 本地层保留高频表达识别
- LLM 补长尾表达
- 最终仍只落到受限选项集里

也就是说：

- 模型可以理解开放语言
- 但不能创造新的关口选项

## 不建议继续扩张本地规则的场景

下面这些信号一旦出现，基本说明应该交给 LLM 了。

### 1. 规则开始覆盖大量近义口语

例如一个意思不断补：

- “该不该做”
- “要不要做”
- “是不是得处理”
- “值不值得做”

如果只是为了补语义同义词，继续堆规则的价值会越来越低。

### 2. 规则开始隐式模拟推理

例如：

- 同时判断角色、动机、组织关系、优先级
- 再拼成一段“像理解过”的结论

这类代码会越来越难维护，也会越来越像假装在推理。

### 3. 新规则很难写成稳定回归

如果一条规则写出来后，很难说明：

- 它解决了什么稳定问题
- 它会不会误伤别的输入
- 它的回归边界在哪里

那它大概率不该继续扩。

## 不建议直接把主线交给 LLM 的场景

反过来，这些能力也不适合完全交给 LLM。

### 1. “让 LLM 决定下一阶段”

不建议直接让模型输出：

- 进入哪个状态
- 哪个关口该阻塞
- 哪类卡片该出现

因为这会让产品主线失去可控性。

### 2. “让 LLM 直接输出完整 case state”

这会导致：

- 状态真相和模型输出混在一起
- 很难判断哪里是系统约束，哪里是模型猜测

### 3. “让 LLM 自由改写决策关口”

关口的本质是产品方法约束，不应在每次调用时自由漂移。

## 边界场景举例

下面用几个具体场景说明如何分层。

### 场景 1：用户说了一句很模糊的话

输入：

- “最近前台总说这里不太顺，我感觉要不要看看。”

建议分层：

- 本地层判断：是否进入 `pre_framing`
- LLM 负责：这句话可能对应的几种理解
- 本地层负责：输出 `继续卡`

### 场景 2：用户补了一句复杂项目背景

输入：

- “这是个 B 端 SaaS，门店店员主要在 H5 上操作，老板会盯结果。”

建议分层：

- 本地规则先提取高频字段
- LLM 补充长尾理解
- 本地层校验并写回 `context_profile`

### 场景 3：用户对决策关口的回答很绕

输入：

- “这事不是完全不做，但现在资源实在紧，可能先别排太前。”

建议分层：

- LLM 负责理解真实倾向
- 本地层将其映射到：`defer`
- 本地层决定是否继续阻塞

### 场景 4：用户已经进入验证设计

建议分层：

- 本地层决定：不再自动回到 `pre_framing`
- LLM 只参与当前阶段的理解增强

也就是说：

- 模型可以增强分析
- 但不能把主线强行拉回前面

## 当前阶段的收敛建议

按当前项目进度，更合理的边界是：

### 本地层优先保留

- 状态机
- 服务层
- 输出契约
- 决策关口
- 高频场景的基础字段抽取
- 自动化测试和回归基线

### LLM 优先补强

- 回复解释器
- pre-framing 候选方向生成
- 卡片文案增强层
- 复杂角色关系抽取
- 更自然的卡片文案
- 长尾口语意图识别

### 近期不建议做

- 用规则继续扩成半套自然语言引擎
- 用 LLM 直接接管全部主线推进

## 一句话总结

`PM Method Agent` 不是要用 LLM 去伪装一套传统规则系统，也不是要把整个产品主线外包给 LLM。更合理的方向是：本地层负责稳定骨架，LLM 负责理解增强，二者通过清晰契约协作。
