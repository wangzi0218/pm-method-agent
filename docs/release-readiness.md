# GitHub 发布标准

## 目标

这份清单用于判断：当前项目是否已经具备第一次公开发布到 GitHub 的条件。

这里说的“可以发布”，不是指项目已经成熟，而是指：

- 外部用户能看懂它是什么
- 外部用户能跑起来
- 外部用户能理解当前边界
- 仓库内容不会给人一种“只有作者自己能用”的感觉

## 建议分级

### 可以先公开的状态

满足下面这些条件，就已经可以公开发到 GitHub：

- README 能说明项目解决什么问题、不是做什么、如何运行
- CLI 能稳定跑通至少 4 到 6 个典型场景
- 输出结构已经基本稳定，不会今天一套明天一套
- 已有最小测试，能覆盖基础 smoke path
- 已明确说明当前阶段仍是方法内核验证，不是最终用户入口

这对应的是一个“早期但可信”的开源状态。

### 更适合公开推广的状态

如果希望不仅发上去，还希望更容易被 star、被理解、被复用，最好再补到下面这个程度：

- 有完整体验用例集
- 有一组可复用样例输出
- 安装方式足够稳，不容易卡在环境问题
- 有清晰的路线图和已知限制说明
- 至少完成一轮真实用户体验反馈后的修订

### 暂时不建议公开的状态

如果还处在下面这些情况，建议先不要推公开仓库：

- README 还是偏内部脑图，外部人读不懂
- 运行入口不稳定，经常需要口头解释才能跑
- 输出结构还在频繁变动，连“卡片长什么样”都没稳定
- 没有样例，也没有测试
- 文档和实际实现明显不一致

## 首次发布的最小门槛

建议至少满足这 8 项：

1. 项目定位清楚  
README 必须让第一次看到仓库的人在 3 分钟内理解项目价值。

2. 运行方式清楚  
至少提供一种稳定可用的运行方式。即使是 `PYTHONPATH=src python3 -m ...`，也可以接受。

3. 输出样式稳定  
当前输出结构和字段含义不能处于频繁重命名阶段。

4. 有体验用例  
至少准备 5 个覆盖不同语境的体验用例。

5. 有示例产物  
至少保留 1 到 2 份示例输出或示例案例。

6. 有最小测试  
至少覆盖 CLI 主路径、场景补充卡、决策关口卡、审查卡。

7. 有边界说明  
明确告诉用户：当前是 CLI 验证版，不是最终的 agent 交付形态。

8. 有下一步路线  
说明后续会如何从单轮 CLI 升级到多轮协作和 agent 入口。

## 当前仓库距离“可公开”的判断

按现在的状态看，这个仓库已经接近“可以先公开”的区间，但还差几件最好补齐的事：

- 把安装/运行说明再做稳一点
- 再做一轮基于真实体验的文案修订
- 最好补一份首次公开前的提交与发布说明

如果这几项补齐，我会建议你就可以先推 GitHub 了。

## 建议的首次公开版本定义

建议把第一次公开发布定义为：

- `v0.1.0`
- 定位：方法内核验证版
- 入口：CLI
- 重点：稳定结构化审查卡，而不是完整 agent 交互

这样的版本定义是诚实的，也比较容易被理解。

## 推上 GitHub 前的最后检查

推送前，至少手动检查一次：

- README 是否和当前输出一致
- 示例命令是否真能跑
- 测试是否通过
- 仓库里没有明显的内部语境残留
- 文案没有大面积中英混杂
- 路线图没有承诺过多尚未实现的能力
