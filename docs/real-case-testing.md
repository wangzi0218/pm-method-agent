# 真实问题试跑指南

## 目标

这份文档不是用来替代自动化测试，而是用来回答一个更实际的问题：

当你把一个真实需求、真实抱怨，或者一个还没想清楚的想法交给 `PM Method Agent` 时，它能不能接得住。

这里关注的不是“代码有没有报错”，而是：

- 能不能识别这是一个问题线索，还是一个已经带方案的表达
- 能不能在信息不够时停下来，而不是硬往下分析
- 能不能随着补充信息继续承接，而不是每轮都像重新开聊
- 输出是不是像一张自然、克制、能继续协作的审查卡

## 和自动化测试的区别

当前仓库里已有的自动化测试，主要验证：

- 状态推进是否正确
- 会话是否能承接
- HTTP 和 agent 入口是否可用
- 基础输出结构是否稳定

真实问题试跑补的是另一层：

- 真实表达能不能被理解
- 判断是否贴近业务场景
- 文案和交互是否顺口
- 哪些地方会误判、漏判或说得太重

这两层都要有，才算一个能持续打磨的产品。

## 当前已补的自动化模拟

除了手工试跑，当前仓库也已经补了一组更贴近真人表达的自动化用例。

可直接看：

- [tests/test_human_like_flows.py](/Users/wannz/Documents/sourcetree/pm-method-agent/tests/test_human_like_flows.py)
- [partial-follow-up-checklist.md](/Users/wannz/Documents/sourcetree/pm-method-agent/docs/partial-follow-up-checklist.md)

这组用例不是只测“有没有报错”，而是刻意模拟几种常见的真人使用方式：

- 初级产品经理：表达模糊，先说感受，再补背景
- 中级产品经理：会直接带部分场景和问题描述
- 高级产品经理：会带上组织语境、替代路径和阶段判断

这组自动化测试当前主要保护三件事：

- 第一轮输入是否能落到合理卡片，而不是一上来就出完整分析
- 多轮补充后，系统是否顺着主线推进，而不是每轮重新开聊
- 同样一句自然表达，系统是否能正确识别场景、角色和决策倾向

这里还有一个测试边界要先说清：

- 不是所有“半步回答”都适合只靠整轮回放来验
- 有些补充一旦说出来，运行时就会合理推进阶段
- 这类场景更适合：
  用真实回放看主线有没有跑偏
  再用模板级回归锁住追问文案

当前比较典型的两类是：

- 角色三分法
- 机会成本

## 适合拿来试跑的输入

建议优先拿下面四类真实表达来测：

### 1. 模糊想法

例如：

- “前台最近老是漏提醒患者，我在想是不是要处理一下。”
- “新用户进来后好像不知道下一步该做什么。”

想看的是：

- 系统会不会先把场景和问题补清楚
- 会不会误把一句模糊抱怨当成完整需求

### 2. 方案先行

例如：

- “想加一个新手引导浮层，提升新用户发帖率。”
- “希望做一个提醒弹窗，减少漏操作。”

想看的是：

- 系统能不能识别“输入里已经带了方案”
- 会不会提醒先回到问题本身

### 3. 指标异常

例如：

- “最近注册后 3 天内发帖率掉了 15%，运营怀疑首帖路径有问题。”
- “本月预约到诊率比上月低很多，门店说提醒环节不稳定。”

想看的是：

- 系统能不能围绕证据、基线和验证来展开
- 不会因为输入更完整，就直接跳过关键追问

### 4. 组织流程问题

例如：

- “审批总是漏人，想补一个抄送能力。”
- “护士交接不稳定，想做一个交接确认页。”

想看的是：

- 系统能不能把非产品路径一起拿出来比较
- 会不会过早默认“做功能”就是答案

## 建议怎么跑

### 路径一：直接测单轮输入

适合先看第一反应是否靠谱。

```bash
PYTHONPATH=src python3 -m pm_method_agent.cli agent \
  --workspace-id real-idea-1 \
  "前台最近老是漏提醒患者，我在想是不是要处理一下。"
```

重点看三件事：

- 第一张卡片停在哪里
- 追问是不是最小必要信息
- 语气是不是自然

### 路径二：测完整多轮互动

适合验证这个项目最关键的能力，也就是“承接”。

第一轮：

```bash
PYTHONPATH=src python3 -m pm_method_agent.cli agent \
  --workspace-id real-flow-1 \
  "前台最近老是漏提醒患者，我在想是不是要处理一下。"
```

第二轮补场景：

```bash
PYTHONPATH=src python3 -m pm_method_agent.cli agent \
  --workspace-id real-flow-1 \
  "这是一个 ToB 移动端产品，前台使用，管理者负责结果。"
```

第三轮补现状与证据：

```bash
PYTHONPATH=src python3 -m pm_method_agent.cli agent \
  --workspace-id real-flow-1 \
  "现在靠前台手工翻列表确认提醒，每天大概会漏 3 到 5 个患者，之前试过晨会提醒，但一周后又回去了。"
```

第四轮补决策信息：

```bash
PYTHONPATH=src python3 -m pm_method_agent.cli agent \
  --workspace-id real-flow-1 \
  "这件事这个月必须处理，因为最近投诉变多了；如果继续靠人工盯，店长每天还要额外花 20 分钟复核。"
```

第五轮补验证标准：

```bash
PYTHONPATH=src python3 -m pm_method_agent.cli agent \
  --workspace-id real-flow-1 \
  "我希望先看漏提醒率能不能下降到 1% 以内，同时不要明显拖慢前台接待效率。"
```

这条路径重点看：

- 卡片会不会随着输入逐步收敛
- 每一轮是否都承接上一次，而不是来回重复
- 原本阻塞的地方，会不会因为信息补齐而自然推进

### 路径四：专门抽查“半步回答”

这条路径不是看系统能不能识别完整回答，而是看：

- 用户只答到一半时，系统会不会顺着那半步继续问
- 下一轮追问会不会比原始结构化问题更像自然对话
- 系统会不会又退回“重新问一遍原题”

推荐至少抽两类：

#### 1. 指标类半步回答

第一轮：

```bash
PYTHONPATH=src python3 -m pm_method_agent.cli agent \
  --workspace-id half-step-metric \
  "这是一个 ToC 内容社区 App，新用户注册后 3 天内发帖率偏低，新用户和内容运营都在关注这个问题，运营怀疑他们不知道首帖该发什么。"
```

第二轮只补半句：

```bash
PYTHONPATH=src python3 -m pm_method_agent.cli agent \
  --workspace-id half-step-metric \
  "我觉得这件事如果能提升一点发帖率就值得看，但具体目标还没想清。"
```

这时重点看：

- 系统有没有识别这是“成功指标只答到一半”
- 追问有没有收窄成类似“做到什么程度，你会觉得这轮值得继续”
- 而不是继续原样问“成功指标是什么”

#### 2. 非产品路径半步回答

第一轮：

```bash
PYTHONPATH=src python3 -m pm_method_agent.cli agent \
  --workspace-id half-step-non-product \
  "最近诊所前台老说提醒总会漏，我在想是不是要处理一下。"
```

第二轮补背景：

```bash
PYTHONPATH=src python3 -m pm_method_agent.cli agent \
  --workspace-id half-step-non-product \
  "这是个 ToB HIS 产品，平时前台在网页端操作，店长盯结果。"
```

第三轮只补半句：

```bash
PYTHONPATH=src python3 -m pm_method_agent.cli agent \
  --workspace-id half-step-non-product \
  "我也在想能不能先靠晨会提醒和流程约束顶一下，但还没比较清楚。"
```

这时重点看：

- 系统有没有识别这是“非产品路径只答到一半”
- 追问有没有收窄成类似“先靠流程或运营能不能兜住一部分”
- 决策关口还在不在，还是被半句输入误放行了

### 路径三：先补项目背景，再测真实问题

适合一个项目里连续测试多个真实需求。

先建立项目背景：

```bash
curl -X POST http://127.0.0.1:8000/project-profiles \
  -H "Content-Type: application/json" \
  -d '{
    "project_name": "医疗服务平台",
    "context_profile": {
      "business_model": "tob",
      "primary_platform": "mobile-web",
      "target_user_roles": ["前台", "诊所管理者"]
    },
    "stable_constraints": ["上线周期紧", "必须兼容老门店流程"]
  }'
```

再通过统一入口发消息：

```bash
curl -X POST http://127.0.0.1:8000/agent/messages \
  -H "Content-Type: application/json" \
  -d '{
    "workspace_id": "clinic-demo",
    "message": "前台最近老是漏提醒患者，我在想是不是要处理一下。"
  }'
```

这条路径重点看：

- 默认项目背景有没有被正确带入
- 多个真实问题之间，背景是否复用得当

## 试跑时怎么判断结果好不好

建议至少看下面六项。

### 1. 是否接住原始表达

好的表现：

- 模糊表达不会被误判成完整需求
- 抱怨、线索、草稿也能顺着推进

不好的表现：

- 上来就给完整方案建议
- 没补信息就开始下结论

### 2. 是否把方案和问题分开

好的表现：

- 能指出“输入里已经带方案”
- 但不会每次都机械重复这句话

不好的表现：

- 明明是问题描述，也还在硬说“你带方案了”
- 或者明明是方案先行，却完全没指出

### 3. 是否知道什么时候该停

好的表现：

- 场景没说清时会先补
- 决策没做时会停在关口

不好的表现：

- 一路放行到最后
- 或者每一轮都过度阻塞

### 4. 是否能给出有用的下一步

好的表现：

- 下一步是可以执行的
- 顺序基本合理

不好的表现：

- 只有空泛建议
- 或者一步塞太多

### 5. 半步回答后会不会继续顺着问

好的表现：

- 能看出来用户这轮已经碰到关键点
- 下一轮会优先追这个半步，而不是换题
- 问法会更像“顺着刚才那句话继续”

不好的表现：

- 明明只差半步，却像重新发了一张问卷
- 或者完全没接住，直接换成别的问题

### 5. 语气是否自然

好的表现：

- 像专业协作，不像报告腔
- 能看懂，但不生硬

不好的表现：

- 太像咨询报告
- 太像闲聊或口水话

### 6. 是否保留场景差异

好的表现：

- ToB / ToC / PC / 移动端会有不同提醒
- 流程问题和增长问题会有不同判断重点

不好的表现：

- 所有问题都像同一张卡套出来的

## 建议怎么记录

每次真实问题试跑后，至少记这几项：

- 原始输入是什么
- 所属场景是什么
- 哪一句判断最准确
- 哪一句读起来最别扭
- 哪个判断明显缺失
- 下一轮最应该改哪里

## 当前阶段的通过标准

在这个阶段，真实问题试跑不要求“判断完全正确”，但至少应满足：

- 第一轮能接住真实表达，不明显跑题
- 多轮补充后，卡片会逐步收敛
- 阻塞和放行大体合理
- 用户能自然知道下一步怎么继续
- 同一类问题多跑几次，输出结构仍然稳定

## 之后可以怎么升级

当真实问题试跑积累到一定数量后，可以继续升级成一套更稳定的评测机制：

- 把高频真实问题整理成固定评测集
- 给每类问题定义期望输出特征
- 把“语气、误判、缺失判断”也纳入回归检查

到那一步，这个项目就不只是“能跑”，而是开始具备稳定打磨产品体验的基础。
