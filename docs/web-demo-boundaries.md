# 网页 Demo 边界说明

这份文档只回答一个问题：

`如果现在开始做 PM Method Agent 的网页 demo，它到底负责什么，不负责什么。`

它不是前端技术方案，也不是视觉稿。

它更像一份收口说明，用来避免第一版网页 demo 一上来就被做成：

- 一个看起来什么都想接的平台
- 一个默认能替用户执行各种本地动作的网页
- 一个边界不清、后面很难维护的半成品

## 一句话定位

第一版网页 demo 更适合被定义成：

- 一个本地可运行的轻网页壳
- 一个围绕“发消息、看卡片、继续补充、切案例”展开的交互入口
- 一个复用现有本地 HTTP 服务的方法层前端

它当前不是：

- 完整托管平台
- 团队协作后台
- 通用 AI 聊天网页
- 本地开发代理控制台

## 为什么要先讲边界

网页比 CLI 更容易让人默认产生很多期待。

比如用户会自然觉得：

- 能不能登录后跨设备继续
- 能不能把项目资料都传上来
- 能不能直接在网页里调工具、跑命令、改配置
- 能不能顺手做成一个团队共享工作台

这些方向不是永远不做，而是都不该挤进第一版。

第一版网页 demo 更重要的任务只有两个：

1. 让用户更自然地体验 `PM Method Agent` 的主线分析能力。
2. 验证网页入口是否真的比 CLI 更容易上手、更容易连续使用。

## 第一版面向谁

第一版网页 demo 更适合这几类人：

- 想体验方法能力，但不想长期记 CLI 命令的人
- 想在本地快速跑一个真实需求草稿的人
- 想验证网页入口是否适合后续做成 IDE 外壳、桌面壳或托管产品的人

它暂时不优先服务这些诉求：

- 团队级协作与共享沉淀
- 组织级权限和审计
- 重度本地工程操作
- 面向运营后台的复杂配置

## 默认部署边界

第一版网页 demo 默认按下面这个前提成立：

- 网页运行在本地
- 后端服务运行在本地
- 数据默认存在本地 store
- 一个浏览器页面对应一个本地用户环境

更具体一点说：

- 网页 demo 默认连接本机的 `HTTP` 服务
- 网页本身不重写分析逻辑，只消费现有服务接口
- CLI、agent、网页 demo 共用同一套方法内核、状态对象和运行时

这意味着第一版更像：

- `本地服务 + 本地网页壳`

而不是：

- `云端账号系统 + 托管网页平台`

## 能力范围

第一版网页 demo 建议只承接这些核心能力。

### 1. 发起分析

用户可以直接输入一句自然语言草稿，例如：

- 一个模糊想法
- 一段用户反馈
- 一个指标异常
- 一个已经带方案倾向的需求描述

网页负责把原话交给工作区消息入口，不负责自己判断该走哪个阶段。

### 2. 展示当前主卡片

网页需要稳定展示当前案例的主输出，包括：

- 当前阶段
- 当前判断
- 当前关键判断
- 当前建议动作

第一版优先直接展示 `rendered_card`，而不是一开始就把所有区块拆成复杂组件。

### 3. 承接多轮补充

用户可以继续补一句话：

- 补场景
- 补角色
- 补证据
- 回答当前关口

网页只负责把补充继续发给同一个工作区，让运行时自己决定如何推进。

### 4. 查看最近案例与切换案例

网页需要支持：

- 看到最近分析过的案例
- 切换当前活跃案例
- 回到上一次没看完的案例

这是网页入口比纯 CLI 更自然的一部分，也是第一版值得验证的体验点。

### 5. 查看历史与当前阻塞

网页需要能看到：

- 当前案例历史
- 当前工作区运行态摘要
- 当前是否存在待确认关口
- 当前是否存在待处理审批

但第一版只需要做到“看得见、能继续”，不需要做成完整工作台。

这里的运行态展示更适合是：

- 当前卡在哪一步
- 最近恢复点在哪里
- 有没有走模型回退

而不是把网页做成一个工程调试面板。

## 第一版暂不承担的能力

下面这些能力，即使以后可能会做，第一版也不应默认承诺。

### 1. 团队协作

暂不支持：

- 多人同时查看同一工作区
- 团队共享案例池
- 评论、指派、协作流转
- 组织级权限模型

### 2. 云端账号与托管能力

暂不支持：

- 注册登录
- 云端同步
- 跨设备恢复
- 托管多租户

### 3. 完整配置后台

暂不支持：

- 大而全的项目设置中心
- 可视化规则管理后台
- 模型配置面板
- 复杂审批策略编辑器

第一版更适合沿用本地配置和已有文档入口。

### 4. 本地危险动作的网页化入口

暂不默认支持：

- 在网页里直接执行本地命令
- 在网页里直接读写任意本地文件
- 把运行时工具链完整暴露给普通用户

原因很直接：

- 这会把网页入口从“分析产品”变成“本地代理控制台”
- 会明显抬高安全和权限边界
- 会让第一版的产品心智变得混乱

### 5. 通用 AI 聊天能力

网页 demo 不是为了变成一个“什么都能聊”的聊天页。

它更聚焦在：

- 问题定义
- 决策挑战
- 验证设计
- 相关的多轮承接

如果用户只是想自由闲聊，网页 demo 不需要为这种场景做过多设计。

## 服务边界

网页 demo 与服务层的关系应保持简单：

- 网页只消费现有 HTTP 接口
- 服务层继续负责工作区、案例、阶段推进、审批和运行时状态
- 方法判断仍然由内核负责，不下放到前端

第一版前端不应自己维护这些逻辑：

- 阶段状态机
- 决策关口推进规则
- 证据充分度判断
- 方法模块路由

前端应该做的事更少一些：

- 发送消息
- 渲染返回结果
- 切换当前案例
- 展示历史和阻塞提示

## 数据边界

第一版网页 demo 默认按本地数据心智处理：

- 工作区数据默认留在本地
- 案例历史默认留在本地
- 项目背景默认留在本地 store

这也意味着当前不默认承诺：

- 云端备份
- 在线同步
- 数据共享
- 多端一致性

如果后面进入托管版，这部分边界要单独重讲，不能直接沿用本地网页 demo 的默认心智。

## 安全边界

第一版网页 demo 需要明确克制：

- 不默认拥有用户机器上的更高权限
- 不默认越过本地运行时策略
- 不绕开审批、hook 和执行账本

换句话说：

- 即使网页未来接更多能力，也应该继续走统一 runtime
- 不能因为“有个按钮更方便”就绕开底层限制

## 和其他入口的关系

网页 demo 不是新的产品内核，只是新的入口形态。

它和现有入口的关系可以这样理解：

- `CLI agent`：最适合做调试、回归和深度体验
- `HTTP service`：当前最稳定的统一服务底座
- `网页 demo`：最适合做更低门槛的本地体验入口
- `IDE / skill`：更适合深度嵌入用户已有工作环境

所以网页 demo 的正确位置不是替代它们，而是补一层：

- 更容易试用
- 更容易演示
- 更容易让非开发用户理解

## 第一版成功标准

如果网页 demo 要算第一版站住，至少应满足这些判断：

1. 用户不看内部文档，也能完成一轮真实输入和一轮继续补充。
2. 用户能看懂当前卡片，并知道下一步该补什么。
3. 用户能切回最近案例，而不是每次都像重新开始。
4. 网页没有自行发明第二套状态机或第二套方法判断。

## 对后续实现的约束

这份边界文档落下后，后面的真实网页实现建议继续遵守三条约束：

1. 先做轻网页壳，不先做重平台。
2. 先把主链路跑顺，不先做配置中心。
3. 先复用现有服务层，不重写方法内核。

## 当前结论

所以，第一版网页 demo 的合理定义应该是：

- 一个本地可运行的轻交互壳
- 一个围绕主线分析体验的网页入口
- 一个复用现有 HTTP 服务的前端层

只要先把这个边界守住，后面无论继续往：

- 本地桌面壳
- IDE skill
- 托管网页
- 混合架构

去演进，都会更稳一些。
