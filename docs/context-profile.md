# 场景基础信息

## 为什么需要这层定义

`PM Method Agent` 不应该假设所有需求都处在同一种产品语境里。

下面这些维度会显著改变方法判断：

- 企业产品、消费者产品，还是内部系统
- 桌面端、移动端、小程序，还是多端协同
- 私有部署、软件服务化交付、公域分发，还是组织内部系统
- 面向老板、运营、一线员工，还是普通消费者
- 是否存在强合规、硬件、交付周期或组织资源约束

如果这些前提不清楚，系统很容易在错误的上下文里给出“看起来有道理”的结论。

## 最小字段建议

建议先支持下面这几类字段：

- `business_model`
- `primary_platform`
- `distribution_channel`
- `product_domain`
- `target_user_roles`
- `constraints`

其中最优先的是：

1. `business_model`
2. `primary_platform`
3. `target_user_roles`

因为这三项最直接决定：

- 价值判断的逻辑
- 交互空间与渠道限制
- 关键角色结构与冲突关系

## 场景差异举例

### 企业产品 vs 消费者产品

企业产品场景下，很多问题不能只看终端体验，还要看：

- 组织流程是否可执行
- 权限与责任链是否清楚
- 采购方、使用方、管理方是否一致

消费者产品场景下，更常见的关注点是：

- 使用门槛
- 触达效率
- 转化路径
- 用户是否会自然流失

### 桌面端 vs 非桌面端

桌面端产品通常更适合：

- 高信息密度
- 多任务切换
- 复杂配置和批量操作

移动端、小程序或其他非桌面端场景更容易受到：

- 展示空间受限
- 输入效率低
- 中断频繁
- 渠道入口浅

所以某些在桌面端成立的方案，在移动端可能完全不成立。

## 推荐交互策略

不要一开始让用户填写冗长表单，而应采用渐进补充：

1. 先接收原始问题描述
2. 判断是否缺少关键场景信息
3. 只在会明显影响判断时追问最小字段
4. 把补充结果写回场景基础信息

## 一个建议的追问顺序

当场景信息不足时，优先追问：

1. 这是企业产品、消费者产品还是内部产品？
2. 主要使用端是桌面端、移动端、小程序，还是多端？
3. 谁提这个需求，谁使用它，谁承担结果？
4. 有没有明显的约束，比如合规、设备能力、交付周期？

## 在系统中的位置

建议把场景基础信息放在方法分析之前，但不要把它做成一个独立的重阶段。

更合适的做法是：

- 把它视为“阶段 0：场景对齐”
- 由编排器决定是否需要触发
- 一旦信息足够，就进入后续方法分析
