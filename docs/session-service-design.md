# 多轮会话与服务层设计

## 目标

这一阶段要解决的问题，不是继续增加更多分析模块，而是让系统真正具备“接住用户回答并继续推进”的能力。

当前 CLI 已经能完成单轮分析，但还存在一个明显缺口：

- 系统能提出要补什么
- 用户也可以回答
- 但回答之后，系统还不能自然承接上一轮继续往下走

这一阶段的目标，就是补上这层能力。

## 这一阶段要解决什么问题

从用户体验上看，当前流程还是：

1. 用户输入原始需求
2. 系统输出一张卡
3. 用户自己理解缺什么
4. 用户自己拼出下一次输入
5. 再重新跑一遍

这适合验证方法，但还不够像一个真正的协作产品。

下一阶段希望变成：

1. 用户输入原始需求
2. 系统输出当前卡片，并明确追问最关键的内容
3. 用户直接回答这些问题
4. 系统把回答写回状态
5. 系统只从被阻塞的阶段继续推进
6. 直到进入下一阶段、下一关口或结束当前轮次

## 一句话定义

下一阶段的核心，不是“网页”也不是“开放 API”，而是：

`把当前单轮分析器升级成可承接多轮状态的服务系统。`

## 设计原则

### 1. 先补服务层，再补外壳

不要先做网页，再临时想办法接状态。

更稳的顺序应该是：

- 方法核心
- 服务层
- CLI 会话模式
- 本地 HTTP 服务
- 主代理 / skill 入口
- 网页入口
- 对外 API 或 MCP

### 2. 用户直接回答，不重写整段输入

用户回答时，应当可以直接自然语言补充，而不是被迫重写完整 prompt。

### 3. 只从当前阻塞点继续，不要整轮重跑

如果系统已经完成了问题定义，只是在决策挑战阶段被阻塞，就应从那里继续，而不是每次都重新走完整流程。

### 4. CLI、网页、agent 共用同一套状态承接逻辑

不要为 CLI、网页和主代理分别写三套“继续分析”逻辑。

## 服务层的职责

服务层建议放在方法核心和交付入口之间，至少承担这些职责：

- 创建 case
- 读取 case
- 更新 case 的场景信息和补充回答
- 判断当前从哪个阶段继续
- 调用 orchestrator / analyzer
- 保存阶段历史
- 返回当前最新卡片

换句话说，服务层不是新分析器，而是“分析流程的状态管理器”。

## 推荐结构

建议未来演进成：

### `core`

负责：

- `CaseState`
- 分析模块
- 阶段状态机
- 决策关口
- 输出渲染契约

### `service`

负责：

- case 创建与读取
- 用户补充信息合并
- 阶段继续执行
- 历史记录
- 对外提供统一调用接口

### `interfaces`

负责：

- `cli`
- `agent`
- `web`
- `http api`

## 多轮会话的最小交互模型

### 场景 1：补场景信息

当前卡片：

- 缺产品类型
- 缺主要平台
- 缺关键用户角色

用户回答：

```text
这是一个面向诊所前台的 ToB 移动端产品，主要由前台使用，管理者负责结果。
```

系统动作：

- 提取结构化场景信息
- 写回 `case state`
- 从 `context_alignment` 继续进入 `problem_definition`

### 场景 2：补现状证据

当前卡片：

- 缺失败案例
- 缺现状流程
- 缺影响范围

用户回答：

```text
现在前台是手动在预约前一天翻表提醒，最近两周漏提醒了 6 次，主要发生在高峰期。
```

系统动作：

- 把这段回答记录为补充材料
- 更新 evidence / unknowns
- 从 `problem_definition` 或 `decision_challenge` 继续

### 场景 3：做决策关口选择

当前卡片：

- 系统建议先评估非产品路径

用户回答：

```text
我们已经试过培训和流程提醒，两周后效果就回落了，还是想继续产品化。
```

系统动作：

- 记录用户决策
- 更新 gate 结果
- 进入 `validation_design`

## 会话状态建议新增的字段

当前 `CaseState` 已能支持单轮分析，但进入多轮模式后，建议补这些字段：

- `case_id`
- `workflow_state`
- `current_stage`
- `stage_history`
- `conversation_turns`
- `latest_user_reply`
- `collected_facts`
- `answered_questions`
- `resolved_gates`
- `render_version`

### `conversation_turns`

用于记录完整交互历史，建议每条至少包含：

- `turn_id`
- `role`
- `content`
- `turn_kind`
- `created_at`

### `stage_history`

用于记录阶段推进轨迹，建议每条至少包含：

- `from_stage`
- `to_stage`
- `trigger`
- `timestamp`

### `resolved_gates`

用于记录用户已经做过的关口决策，避免重复询问。

## 建议的服务接口

无论最终是 Python 本地服务还是 HTTP API，建议先抽象出这些动作：

### `create_case`

输入：

- 原始问题描述
- 初始场景信息

输出：

- `case_id`
- 最新 `case state`
- 第一张用户可见卡片

### `reply_to_case`

输入：

- `case_id`
- 用户补充回答

输出：

- 更新后的 `case state`
- 新卡片
- 当前是否仍被阻塞

### `get_case`

输入：

- `case_id`

输出：

- 当前完整状态
- 最新卡片

### `get_case_history`

输入：

- `case_id`

输出：

- 当前会话历史
- 阶段推进信息
- 已处理关口

### `rerun_case_stage`

输入：

- `case_id`
- 指定阶段
- 可选的新上下文

输出：

- 更新后的状态

## CLI 在下一阶段应如何演进

CLI 不需要被废弃，反而很适合先承接多轮模式。

建议增加两类能力：

### 1. 会话模式

例如：

```bash
pm-method-agent start "前台希望增加一个预约前提醒弹窗，避免漏提醒患者。"
```

输出：

- `case_id`
- 当前卡片

### 2. 继续回答

例如：

```bash
pm-method-agent reply case-001 "这是一个 ToB 移动端产品，前台使用，管理者负责结果。"
```

输出：

- 更新后的卡片

这样可以先在 CLI 里验证多轮交互，再迁移到网页和 agent。

## 网页与 API 在这一阶段的关系

网页不是先于服务层的事情，而是服务层的一个使用者。

当服务层具备 `create / reply / get / rerun` 这些动作后：

- CLI 可以直接调用服务层
- 网页可以通过 HTTP API 调服务层
- agent 入口也可以调同一层

也就是说，这一阶段真正要做的是“统一调用内核”，不是“单独做一个网页版本”。

## 建议的实现顺序

### 第一步：抽出服务层接口

目标：

- 不改 analyzer 逻辑
- 把当前 `run_analysis_with_context` 包装成 service action

### 第二步：补会话存储

MVP 可以先从最轻的方式开始：

- 本地 JSON 文件
- 简单目录式 case store

先不急着上数据库。

### 第三步：让 CLI 支持 `start / reply / show`

目标：

- 先把多轮协作流程跑顺

### 第四步：补网页或 HTTP API

目标：

- 把同一套服务动作暴露给网页或外部入口

## 当前阶段的非目标

这一阶段暂时不做：

- 复合多代理执行
- 复杂权限系统
- 正式生产级数据库设计
- 对外开放平台能力

## 完成标志

如果下面这些条件同时满足，就说明这一阶段基本完成：

- 用户可以在上一轮卡片基础上直接回答
- 系统可以承接回答继续推进
- 不需要每次重写完整需求
- CLI 和未来网页都能复用同一套服务动作
